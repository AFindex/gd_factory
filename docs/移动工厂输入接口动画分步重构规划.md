# 移动工厂输入接口动画分步重构规划

## 1. 这次重构的目标

输入接口当前的问题不是单个速度参数不对，而是“视觉宿主”和“逻辑阶段”混在一起了：

- 世界传送带上的大包还没形成清晰的交接感，就已经开始播放整套接口动画。
- 同一个大包会在世界侧、桥位、舱内侧、解包舱侧之间出现跳宿主、闪现、补播的问题。
- 逻辑上的缓存位、桥接位、处理位和表现上的锚点没有做到一一对应。

所以这次不从微调入手，而是先把输入接口的动态表现全部清空，回到纯功能基线，然后按阶段逐步恢复。

## 2. 当前基线

本轮代码已经把 `InputPort` 的动态表现关闭，当前应当满足：

- 输入接口仍然保留功能逻辑，世界大包仍可进入移动工厂。
- 输入接口只保留静态建筑模型，不再显示世界侧缓存包、桥接包、舱内缓存包。
- 输入接口不再播放脉冲灯和内部重货动态展示。
- 解包舱仍然保留自己的处理逻辑，但大包的接舱位/处理位展示也暂时关闭，避免它在接口口门附近造成“像接口还在闪”的错觉。

这个基线的目的只有一个：先确认“没有任何输入接口动态演出时，功能链路是稳定的”。

## 3. 重建原则

后续重建时，必须坚持以下原则：

- 一个阶段只允许一个视觉宿主。
- 一个大包在同一时刻只允许出现一次。
- 逻辑缓存和视觉缓存必须一一对应，不能用一个逻辑位驱动两个画面。
- 每恢复一个阶段，只做这一小段，不顺带恢复下一段。
- 每一阶段做完后先由你目测验收，再进入下一阶段。

## 4. 目标链路

最终希望输入接口形成下面这条清晰链路：

1. 世界传送带把大包送到世界侧入口前。
2. 大包从世界传送带自然滑入世界侧入口缓存位。
3. 大包在世界侧缓存位短暂停留，明确表现“接口已接货”。
4. 大包从世界侧缓存位滑到桥位。
5. 大包穿过桥位进入舱内侧缓存位。
6. 大包在舱内侧缓存位等待解包舱接货。
7. 解包舱接货时，大包从舱内缓存位移动到解包接舱位。
8. 大包进入解包处理位，开始处理动画。
9. 处理结束后大包淡出，小包再从解包舱吐出。

## 5. 建议的分步实现顺序

### 第 0 步：纯静态基线

目标：

- 输入接口完全没有动态货物表现。

验收点：

- 不应再看到世界侧、桥位、舱内侧出现任何大包闪现或补播。
- 即使功能仍在流转，输入接口本体也只是一座静态建筑。

### 第 1 步：只恢复“世界传送带 -> 世界侧入口缓存”

目标：

- 只做第一段。
- 大包从世界传送带末端滑进世界侧入口缓存位。
- 一旦进入缓存位，世界传送带上的物体就正式消失。

本步不做：

- 不做桥接。
- 不做舱内显示。
- 不做解包舱联动。

验收点：

- 只能看到一段世界侧接货滑动。
- 大包最终停在世界侧缓存位，不会继续往里走。

### 第 2 步：恢复“世界侧缓存停留”

目标：

- 让大包在世界侧缓存位停留一小段固定时间。
- 这段时间只表达“接口已接货，准备入舱”。

本步不做：

- 不做桥位穿越。

验收点：

- 大包停在世界侧缓存位时，不会闪、不复制、不跳到舱内。

### 第 3 步：只恢复“世界侧缓存 -> 桥位 -> 舱内缓存”

目标：

- 把入舱过程拆成两段连续空间路径：
  - 世界侧缓存位 -> 桥位入口
  - 桥位入口 -> 舱内缓存位
- 这一步只恢复输入接口自己的完整入舱过程。

本步不做：

- 不和解包舱对接。
- 到达舱内缓存后就停住。

验收点：

- 整条路径必须是连续单物体移动。
- 桥位过程里不允许世界侧保留一份，舱内再生成一份。

### 第 4 步：恢复“舱内缓存等待”

目标：

- 大包到达舱内缓存位后，明确停住并等待。
- 舱内缓存位是解包舱接货前的唯一宿主。

验收点：

- 解包舱未准备好时，大包稳定停在舱内缓存位。
- 不会提前跳进解包舱。

### 第 5 步：恢复“舱内缓存 -> 解包接舱位”

目标：

- 当解包舱可接货时，大包从输入接口舱内缓存位移动到解包舱的接舱位。
- 这里只恢复交接，不做处理腔内部动画。

验收点：

- 输入接口的大包离开后，解包舱接舱位才出现该大包。
- 两边不能同时持有同一包。

### 第 6 步：恢复“解包接舱位 -> 解包处理位 -> 淡出”

目标：

- 大包进入解包处理位。
- 处理完成后大包淡出，再开始小包输出。

验收点：

- 大包的消失发生在处理位，而不是接口位。
- 小包输出应当晚于大包淡出，不要重叠得太乱。

## 6. 每一步推荐的实现方式

为了避免再次回到“逻辑和表现纠缠”的状态，建议每一步都按下面顺序做：

1. 先只加一个新的视觉状态枚举，不改其他阶段。
2. 再只加这个阶段对应的唯一锚点。
3. 再只加这段路径的插值。
4. 最后再把状态切换接到逻辑事件上。

不要在同一轮里同时加入多个锚点、多个阶段和多个宿主。

## 7. 输入接口后续推荐的数据结构

为了后面一步步做，输入接口的表现层最好长期保持下面这几个明确的视觉位：

- `WorldApproach`
  - 世界带末端到世界侧缓存位的滑入段。
- `WorldBuffer`
  - 世界侧缓存位。
- `BridgeIn`
  - 从世界侧缓存进入桥位。
- `BridgeOut`
  - 从桥位进入舱内缓存位。
- `InteriorBuffer`
  - 舱内缓存位。
- `UnpackerDock`
  - 解包舱接舱位。
- `UnpackerProcess`
  - 解包处理位。

这几个名字不一定要现在就写死进代码，但概念上必须先固定。

## 8. 推荐的第一步

下一轮建议只做“第 1 步”：

- 世界传送带末端到世界侧缓存位的滑入。
- 先不做缓存停留后的后续动作。
- 先把“接货时机”和“世界侧唯一宿主”彻底做干净。

只要这一步看起来顺了，后面再一段段加，不会再回到现在这种整条链一起坏的状态。

## 9. 第 1 步已实现的要点

第一步实际落地时，最终保留的实现重点如下：

- 世界侧只保留两个可视宿主：
  - `WorldRouteHandoff`
  - `WorldOuterBuffer`
- `InputPort` 世界侧这一步的空间路径不再凭感觉摆点，而是明确拆成两点：
  - 世界传送带与接口接触点
  - 世界侧缓存位
- 世界侧滑入段只允许沿这两个点之间插值，不允许提前出现桥位或舱内侧表现。
- 舱内桥位、舱内缓存位、解包舱重载展示在第一步都继续隐藏，让逻辑继续跑、表现先收窄。
- `FlowTransportStructure` 里遗留的 `LegacyVisual` 也必须一起清理，否则会出现“明明新的表现已经屏蔽，但旧节点还在闪一下”的假象。
- 解包舱自己的大包展示同样要先关掉，否则接口附近看起来像还有第二份货物。

## 10. 第 1 步得到的经验

第一步过程中，确认了几个非常关键的经验，后面必须继续沿用：

- 不要为了做第一步去冻结整条物流逻辑。
  - 如果直接停掉后续逻辑，世界传送带会卡死，反而看不清问题。
  - 正确做法是只限制“谁能显示”，而不是把整条功能链路一起停掉。
- 世界侧物品朝向不能靠“接口路径方向”来猜。
  - 当前世界传送带的优化渲染路径只传位置，不传单物品旋转。
  - 所以第一步看起来正确的朝向，本质上必须对齐“世界传送带当前实际显示出来的朝向规则”，而不是理论运动方向。
- 视觉宿主和逻辑宿主要分开看。
  - 逻辑上货物可以已经进入下一阶段；
  - 但在当前分步重构里，如果这一阶段还没开放显示，对应宿主就必须保持隐藏。
- 一旦出现“横向对、纵向反”或者“左到右对、右到左反”的现象，优先怀疑坐标系或显示规则不一致，不要继续用统一的 `180°` 补偿去赌。
- 文档里的每一步都应该理解成“新增一个稳定可验收的画面”，不是“顺手把后面半条链也接上”。

目前已经完成：

## 12. 第 2 步已实现要点、经验与待办

第二步目前已经实现了“世界侧缓存停留”，但这一步最大的价值不是停留本身，而是把“阶段验证”从“整条链路一起看”改成了“卡在单一宿主上看”。

本步已确认的实现要点：

- 世界侧缓存位必须是第二步唯一宿主。
- 只要还在验第二步，就不能提前放开桥位和舱内缓存位的显示。
- 第二步的停留逻辑最好先独立成一个非常明确的控制项，不要把它混进第三步的桥接进度判断里。
- 即使后续逻辑仍然可以运行，第二步的画面验证仍然应该被严格限制在“接货后停住”。

本步得到的经验和教训：

- 固定时长停留虽然不是最终方案，但非常适合拿来确认“世界侧缓存是否稳定、是否单宿主、是否不闪不补播”。
- 第二步如果一开始就和第三步的启停条件绑死，会很难分辨问题到底出在缓存停留、桥接启动还是宿主切换。
- 先允许一个“笨但稳定”的中间版本，比一开始追求完整状态机更容易看清问题边界。
- 第二步最重要的不是节奏好不好，而是验证“世界侧缓存这张画面能不能稳定成立”。

第二步仍保留的 TODO：

- 世界侧缓存位不应永久依赖固定秒数结束等待。
- 正确结束条件应当由后续阶段驱动：
  - 舱内缓存位空闲；
  - 第三步桥接链路允许启动；
  - 或更进一步，由舱内接口/解包舱返回“允许继续入舱”的通知。
- 也就是说，第二步最终应该是“等到内侧允许进入”，而不是“时间到了就进入”。

## 13. 第 3 步已实现要点、经验与教训

第三步最终验证通过的核心，不是简单把点连起来，而是把“谁在显示、在哪个坐标系里显示、什么时候切宿主”拆成了明确的分段状态机。

本步已确认的实现要点：

- `世界侧缓存 -> 桥入口` 和 `桥出口 -> 舱内缓存` 必须拆成独立段落，不能用一个粗糙插值一把拉到底。
- 上桥前要先做朝向 lerp，朝向对齐桥之后再开始滑动。
- 过桥后不能直接瞬切到舱内最终落点，而是先在桥尾稳定落位，再做朝向 lerp，最后滑入舱内缓存最终位置。
- 舱内桥尾的位置必须基于真实桥尾世界点换算到舱内局部坐标，不能凭感觉估一个近似点。
- 连续运行时，舱内缓存位不能抢先生成下一份货物，必须等桥接过程真正结束后才接管显示。

本步得到的经验和教训：

- 第三步最容易出问题的不是速度，而是宿主切换顺序。
- 只要宿主切换早了一帧，就会出现“桥上还没过去，舱内先冒出来”的假象。
- 只要位置使用了错误的父节点或局部坐标系，就会出现“路径差不多对，但某一段瞬移或断开”的问题。
- 旋转插值不能偷懒用普通角度差，需要显式走最短角路径，否则不同朝向组合下会出现反向绕圈或 180 度翻转。
- “看起来只差一点”的桥尾断裂，实际上通常意味着阶段边界没有切干净，而不是简单调一个偏移量就能解决。

## 14. 第 4 步已实现要点、经验与教训

第四步真正做出来以后，确认“舱内缓存等待”并不是一句“先别交给解包舱”这么简单，它实际上同时牵扯了宿主所有权、重叠缓存、以及第三步独占释放时机。

本步已确认的实现要点：

- 舱内缓存等待必须明确由输入接口自己持有，不能逻辑上在等、画面上却没有宿主。
- 第四步刚开始时，最稳定的版本仍然是单货物模式：
  - 先保证舱内等待这一份能稳定存在；
  - 再考虑世界侧下一份何时进入。
- 当开始恢复“前一份在舱内等、后一份在世界侧进”的重叠流程时，不能再依赖单一 `TryResolveCurrentPresentation` 结果去决定整套可视。
- 必须允许世界侧缓存和舱内缓存各自独立判断、各自独立显示，否则一放开重叠，必然重新出现“有货但被另一侧顶没”的假象。
- 第三步结束后的最后一小段“滑到舱内缓存终点”更适合作为视觉收尾，而不是整条链路的独占截止点。

本步得到的经验和教训：

- “等待时消失”很多时候不是货真的没了，而是显示所有权被另一侧缓存抢走了。
- 遇到这种问题，优先排查“当前允许同时存在几份逻辑货物、当前展示逻辑是否仍只支持单宿主”，不要先靠调锚点或强行改优先级硬顶。
- 如果第四步还要求世界侧完全不能进下一份，那么验证确实简单；但一旦要做真正的连续流程，就必须尽早把“单宿主显示结构”改成“多位置可并行显示结构”。
- 释放桥接独占的时机非常关键：
  - 如果等到前一份完全滑到舱内终点才释放，后续节奏一定发闷；
  - 更合理的做法是“桥已经让出来了，但前一份最后那点内滑仍可继续播完”。
- 第四步不是一个纯静态阶段，它实际上决定了第五步以前整条链是否有能力做出自然重叠。

## 15. 第 5 步已实现要点、经验与教训

第五步真正落地后，确认“把货交给解包舱”不能只看逻辑调用 `TryAcceptHeavyBundle` 成没成功，关键还是要把“接口放手”和“解包舱接住”之间的视觉所有权切换做干净。

本步已确认的实现要点：

- 输入接口与解包舱之间必须明确只有一个所有者。
- 输入接口放手以后，解包舱接舱位才应该出现该大包，不能出现两边同时各拿一份。
- 解包舱接舱位不能凭格子中心硬估起点，最好直接使用输入接口真正放手时的实际世界位置去做接舱 approach。
- 解包舱这一步只恢复 `StagingPayloadAnchor`，继续关掉处理位大包展示和后续推进，这样才能单独看清“接舱”是否顺。
- 如果接舱锚点的位置更新晚于 `SyncPayloadVisual`，就会出现“先闪到位、再被回拉”的错觉，所以接舱位的预摆放必须在可视同步前完成。

本步得到的经验和教训：

- 第五步最容易犯的错不是“没接上”，而是“逻辑接上了，但视觉起点和终点不连续”。
- 只传 source cell 而不传真实 source world position，往往会导致接舱位看起来先偏一下再过去，因为格子中心并不等于接口真实放手点。
- 舱内最后一段朝向对齐如果依赖“当前正在滑动的锚点位置”，也容易被自身动画带偏；最终轴向最好回到默认锚点轴线去求。
- 把第五步拆干净以后，才能稳定进入第六步；否则第六步一开，问题会看起来像“处理位不对”，实际上根子还在接舱过渡本身。

## 16. 当前下一步修正

- 第 6 步：恢复“解包接舱位 -> 处理位 -> 淡出”

本轮要求：

- 大包从解包接舱位推进到处理位。
- 处理位展示恢复，并在释放阶段完成淡出。

## 17. 第 6 步当前实现策略与本轮记录

第六步这一轮先不再额外发明一套“重包专用处理状态机”，而是直接复用解包舱现有的输入缓存、处理舱、释放、吐料链路，只把大包的可视宿主一步步重新放开。

本轮已经接入的实现策略：

- 接舱位大包展示保持开启，并沿 `StagingPayloadAnchor -> ProcessingPayloadAnchor` 做完整推进。
- 当接舱推进完成后，大包正式切换到处理位宿主，由 `ProcessingPayloadAnchor` 继续持有。
- 处理位的大包展示重新开启，但处理位的机械细节只在处理进度起来后逐步增强，避免一接舱就立刻“抖起来”。
- 释放阶段仍由解包舱原有时序推进，但大包会先在处理位完成缩放/淡出，再进入小包连续吐出。
- 大包退场完成后，额外插入一个很短的“清舱空窗”，在这段空窗内既不吐后续小包，也不允许下一包重包进入。
- 小包吐出继续复用原 manifest 节奏，不在这一轮额外插入新阶段，以便先看清“处理位是否成立”。

本轮记录的要点和教训：

- 第六步最关键的不是“让处理位有货”，而是保证接舱位推进到处理位时没有第二次起点重建。
- 如果处理位刚启用时就直接套完整机械运动，画面会看起来像刚进舱就瞬间进入工作高潮，因此机械摆幅和 carriage 滑动最好跟随处理进度逐步放大。
- 第六步仍然要继续坚持“先有重包完整退场，再看小包吐出节奏”，不要为了追求热闹把两套表现重新搅在一起。
- 如果目测上觉得“上一包还没走干净，下一包/下一段就进来了”，不要先去调速度，优先给退场阶段补一个明确的清舱占用窗口，把“淡出完成”变成可判断的硬条件。
- 这一轮代码层面已经接线完成，但是否完全顺眼仍然要以后续目测验收为准；如果还有跳宿主或空窗，优先排查接舱末端与处理位起点是否真正共点。

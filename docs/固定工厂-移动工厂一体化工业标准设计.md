# 《固定工厂 / 移动工厂一体化工业标准设计》

版本：`v0.1 草案`  
语言：中文  
适用项目：`D:\Godot\projs\net-factory`

## 1. 文档目的

本文档用于明确 `net-factory` 项目中“固定工厂模式”与“移动工厂模式”的统一设计方向，重点解决以下问题：

- 固定工厂与移动工厂在视觉尺度上的冲突
- 主角可同时进入世界与移动工厂内部时的空间合理性问题
- 内外物流系统是否需要拆成两套底层逻辑的问题
- 内外建筑、物品、物流表现如何既统一玩法又避免违和的问题

本文档的目标不是直接定义代码实现细节，而是形成后续系统设计、内容配置、美术区分、玩法验证的统一依据。

## 2. 核心结论

本方案的核心结论如下：

- 固定工厂与移动工厂应共享同一套底层物流语法。
- 固定工厂与移动工厂不应共享同一套具体建筑目录与具体物品表现。
- 内外差异的本质应被定义为“工业标准不同”，而不是“同一建筑和物品按比例缩放”。
- 世界侧物流运输的是外部包装态或野外工况物品。
- 移动工厂内部物流运输的是内部供料态或舰载工况物品。
- 内外之间通过专门的转换建筑或转换接口完成物品工况切换。
- 主角进入移动工厂内部时，应进入“真实可行走的工业维护空间”，而不是“缩小版世界工厂”。

一句话概括本设计：

**内外共享物流语言，不共享工业目录；资源语义共通，载体工况分流。**

## 3. 设计目标

### 3.1 主目标

- 保留当前项目已经具备的工厂玩法底层价值。
- 让固定工厂与移动工厂形成明确分工，而不是彼此重复。
- 解决内外建筑、传送带、物品大小不一致带来的视觉违和。
- 支持主角同时出现在世界与移动工厂内部，且空间感成立。
- 将“视觉冲突”转化为“工业标准差异”这一正向设计特征。

### 3.2 次目标

- 尽量避免重写底层物流模拟。
- 优先通过配置分流、站点规则、美术区分和少量转换建筑解决问题。
- 让移动工厂内部仍保留真实空间感，而不是退化为纯抽象 UI 视图。
- 为后续增加更多移动工厂 profile、内部布局、前线玩法保留扩展空间。

## 4. 设计原则

### 4.1 保留统一的底层物流语法

以下内容在世界与内部之间应尽量共通：

- 传送逻辑
- 连接逻辑
- 分流/合流逻辑
- 桥接逻辑
- 输入/输出逻辑
- 回压与阻塞逻辑
- 放置、旋转、拆除、预览逻辑
- 蓝图和拓扑规则

### 4.2 分离工业标准，不分裂游戏规则

以下内容应允许在世界与内部之间不同：

- 建筑目录
- 建筑外观
- 建筑占用语义
- 允许处理的物品工况
- 配方目录
- 视觉比例与工业包装方式
- 站点建造限制

### 4.3 主角尺度必须统一

- 主角的人体尺度在世界与内部必须一致。
- 不允许通过“主角缩小/放大”来解释内部空间。
- 必须通过“空间结构设计”与“物流载体工况设计”来调和尺度问题。

### 4.4 内部不是抽象图，而是真实工业空间

- 移动工厂内部不是纯 UI。
- 内部空间应具备真实的舱段感、维护通道、结构支撑与工业设备感。
- 但内部工业设备不应被设计成外部世界设备的简单缩小版。

## 5. 总体架构

本方案将系统分为四层：

### 5.1 第一层：底层共通层

职责：

- 定义统一的物流行为语法
- 定义统一的站点接口
- 定义统一的连接、流动、阻塞、放置和交互规则

结论：

- 这一层尽量不大改
- 世界与内部继续共用

### 5.2 第二层：站点配置层

职责：

- 区分 `World` 与 `Interior` 两类工业站点
- 决定每个站点允许建造哪些建筑
- 决定每个站点允许哪些工艺和交互规则附加限制

结论：

- 这一层主要通过目录和规则配置完成分流

### 5.3 第三层：物品工况层

职责：

- 保留资源语义一致
- 允许物流载体和包装态不同
- 解决内外物品比例与视觉表现冲突

结论：

- 同一资源在世界与内部可采用不同运输形态

### 5.4 第四层：转换层

职责：

- 在世界与内部之间完成物品工况转换
- 建立两套工业标准之间的桥梁
- 为玩法提供“解包 / 封包 / 缓冲 / 节拍整形”机制

结论：

- 这一层通过少量专门建筑或端口行为实现
- 不要求重写整套物流系统

## 6. 站点类型定义

### 6.1 World 站点

定义：

- 世界地图上的固定网格工业站点
- 用于建设固定工厂、前哨站、采集区、重型防线、接收站和长期物流网络

特征：

- 占地大
- 建筑重型
- 外部环境适配明显
- 可长距离扩张
- 适合主干物流和长期生产链
- 运输物品以外部包装态或野外工况为主

### 6.2 Interior 站点

定义：

- 移动工厂内部的工业舱段站点
- 用于建设舰载/车载模块化工业系统

特征：

- 空间受限
- 高密度、高封装
- 设备与物流嵌入结构
- 适合短链、高功能密度、前线工业行为
- 运输物品以内部供料态或舰载工况为主

## 7. 底层共通需求

以下需求适用于世界与内部：

### 7.1 物流拓扑共通

- 直线运输结构必须遵守统一连接语义。
- 分流器与合流器必须遵守统一端口语义。
- 物流桥接结构必须遵守统一跨越规则。
- 输入口与输出口必须遵守统一输入输出方向语义。
- 回压、阻塞和可达性判断应基于同一逻辑。

### 7.2 放置与交互共通

- 建筑放置应统一支持：选择、旋转、预览、确认、取消。
- 建筑拆除应统一支持：单点拆除、框选拆除。
- 蓝图应统一支持：捕获、预览、应用、校验。
- 同类底层行为应尽量复用 footprint、topology、simulation 逻辑。

### 7.3 站点接口共通

- 世界站点与内部站点应共享同类站点抽象接口。
- 物流发送、建筑查询、边界校验、结构刷新应统一走站点能力接口。
- 后续新增更多站点类型时，应优先复用该接口体系，而不是复制逻辑。

## 8. 站点配置需求

### 8.1 建筑目录分流

系统必须支持至少两套建造目录：

- `WorldBuildCatalog`
- `InteriorBuildCatalog`

### 8.2 目录能力

每个目录应能定义：

- 可建造建筑列表
- 建筑分类
- 建筑显示名称
- 建筑图标
- 建筑美术变体
- 建筑放置限制
- 建筑可接受物品工况
- 建筑可输出物品工况
- 建筑允许出现的站点类型

### 8.3 World 建筑方向

世界建筑应包含但不限于：

- 重型传送带
- 世界分流器
- 世界合流器
- 世界跨桥
- 世界采矿机
- 世界熔炉
- 世界组装机
- 世界仓储
- 大型 depot
- 世界炮塔
- 世界电力设施

### 8.4 Interior 建筑方向

内部建筑应包含但不限于：

- 内部供料轨道
- 内部分料器
- 内部汇流器
- 内部缓冲柜
- 内部精炼模块
- 内部制造模块
- 内部弹药模块
- 内部电力母线节点
- 解包机
- 封包机
- 边界接口模块

### 8.5 建筑身份原则

必须明确：

- 内部建筑不是世界建筑的缩小版
- 内部建筑是同类工业功能的舰载/车载实现版本
- 世界建筑是野外/固定设施实现版本

## 9. 物品工况需求

### 9.1 资源语义与工况分离

系统中的“资源是什么”与“它以什么形态流动”必须解耦。

建议基础结构包含两层语义：

- `ResourceKind`
- `CargoForm`

### 9.2 资源语义示例

资源语义包括但不限于：

- IronOre
- CopperOre
- Coal
- IronPlate
- CopperWire
- Ammo
- MachinePart
- RepairKit

### 9.3 货物工况示例

建议第一阶段至少支持以下三类：

- `WorldBulk`
- `WorldPacked`
- `InteriorFeed`

必要时可扩展：

- `InteriorAmmoFeed`
- `Containerized`
- `Canister`
- `Magazine`
- `Pallet`

### 9.4 基本规则

- 同一 `ResourceKind` 可对应多种 `CargoForm`。
- 世界侧物流默认运输 `WorldBulk` 或 `WorldPacked`。
- 内部物流默认运输 `InteriorFeed`。
- 内部建筑默认不能直接消费世界工况物品。
- 世界建筑默认不能直接消费内部工况物品。
- 跨工业标准流转必须经过转换层。

### 9.5 视觉要求

- 同一资源在不同工况下应保持可识别的共同特征。
- 内外工况之间必须有明显体积、包装、载体差异。
- 颜色、标识、轮廓提示应保留资源可读性。
- 不允许让玩家误以为“同一个裸物体被偷偷缩小了”。

## 10. 转换层需求

### 10.1 目标

转换层用于在 `World` 与 `Interior` 两个工业标准之间建立明确过渡。

### 10.2 第一阶段推荐转换建筑

建议第一阶段实现以下三类：

- `IntakeUnpacker`
- `ExportPacker`
- `TransferBuffer`

### 10.3 IntakeUnpacker

定义：

- 将外部包装态或野外工况物品转换为内部供料态

输入：

- `WorldBulk`
- `WorldPacked`

输出：

- `InteriorFeed`

用途：

- 世界物流进入移动工厂内部的标准入口处理模块

### 10.4 ExportPacker

定义：

- 将内部供料态产物重新转换为世界可运输工况

输入：

- `InteriorFeed`

输出：

- `WorldPacked`

用途：

- 内部产线输出到世界侧 depot、接收站、防线补给线时的标准出口处理模块

### 10.5 TransferBuffer

定义：

- 用于节拍缓冲、工况切换前后暂存、避免输入输出直接硬碰撞

用途：

- 作为解包与封包流程中的缓冲环节
- 让系统在玩法上更稳定
- 为美术与反馈提供合理的中转结构

### 10.6 端口职责

`InputPort` 与 `OutputPort` 除了连接站点，还应承担以下语义之一：

- 工业标准边界接口
- 包装态切换边界
- 物流域切换接口

## 11. 视觉设计需求

### 11.1 世界侧视觉标准

关键词：

- 粗重
- 外露
- 抗环境
- 设施化
- 可维护
- 基建感

要求：

- 建筑应具有明显外部结构壳体
- 传送带应体现露天、支架、厚重输送感
- 仓储应体现箱体、料仓、堆场或 depot 感
- 电力设施应体现外部供电节点特征
- 防御设施应体现地面炮座或固定炮塔感

### 11.2 内部视觉标准

关键词：

- 高密度
- 舱段化
- 嵌入式
- 模块化
- 可维护但压缩
- 工艺供料感

要求：

- 内部物流应优先表现为槽道、轨道、嵌入式输送、供料导轨
- 内部机器应优先表现为模块舱、嵌入式设备、检修口、面板与结构块
- 内部仓储应优先表现为料架、抽屉仓、立式缓存柜
- 内部电力节点不应继续沿用世界电线杆的视觉语义
- 内部炮塔应更接近硬点、炮井、伸缩炮座，而不是小号地面炮塔

### 11.3 禁止项

- 禁止把世界建筑整体缩小后直接放入内部站点
- 禁止让内部与外部传送带仅通过缩放区分
- 禁止通过简单缩小同一物品模型来表示内外工况差异

## 12. 主角空间与内部可进入需求

### 12.1 主角尺度统一

- 主角在世界与内部使用同一人体尺度
- 主角不能因为进入内部而缩放
- 摄像机与空间感应支持该统一尺度

### 12.2 内部空间结构

移动工厂内部应被设计为真实可进入工业舱段，建议采用三层结构语义：

- 人员维护层
- 模块设备层
- 物流输送层

### 12.3 人员维护层

- 主角主要活动在维护通道、猫道、检修平台、格栅通路
- 行走空间需与正常人体尺度匹配
- 玩家与物流不应长期处于同一平面竞争可见性

### 12.4 模块设备层

- 玩家主要看到的是设备维护面、控制面、外露接口、检修仓门
- 机器可部分嵌入地板、墙体、结构骨架

### 12.5 物流输送层

- 物流可位于地板槽道、侧边导轨、墙内管路、吊顶输送线
- 物流尺寸允许比世界侧小，但必须通过“包装工况不同”解释
- 物流不应让玩家产生“小人国微缩工厂”的感受

## 13. 玩法定位需求

### 13.1 固定工厂定位

固定工厂负责：

- 长期产线
- 大规模物流主干
- 稳定仓储
- 大发电与大电网
- 重型加工
- 主基地防御
- 大范围地图占领与沉淀

### 13.2 移动工厂定位

移动工厂负责：

- 前线采集
- 临时加工
- 中继转运
- 弹药与维修补给
- 快速部署
- 局部工业支撑
- 特殊地形或锚点利用

### 13.3 移动工厂不应承担的职责

- 不应成为完整主基地的等价替代
- 不应承担过长、过重、过完整的全产业链
- 不应通过简单扩大内部面积来解决功能焦虑

## 14. 第一阶段 MVP 范围

建议 MVP 只覆盖以下内容：

### 14.1 保持底层物流系统不拆分

- 继续共用现有拓扑与流动逻辑

### 14.2 建立两类站点目录

- `WorldBuildCatalog`
- `InteriorBuildCatalog`

### 14.3 引入三类基础货物工况

- `WorldBulk`
- `WorldPacked`
- `InteriorFeed`

### 14.4 引入三类转换建筑

- `IntakeUnpacker`
- `ExportPacker`
- `TransferBuffer`

### 14.5 完成一条最小演示链路

建议演示链路：

- 世界采矿机采矿
- 世界重型 belt 输送
- 接入移动工厂输入口
- 内部解包
- 内部精炼或组装
- 内部封包
- 输出到世界 depot 或接收站

### 14.6 完成一套基础视觉区分

- 世界侧重型 belt 与内部供料轨有明显区别
- 世界矿物与内部供料盒有明显区别
- 世界熔炉与内部精炼模块有明显区别

## 15. 设计验收标准

满足以下条件，可视为本方案第一阶段成立：

- 玩家能理解固定工厂与移动工厂的分工差异
- 玩家不会再把内部设备理解为外部设备的缩小版
- 玩家能接受同一资源在内外物流中表现不同
- 玩家进入移动工厂内部时不会产生明显“小人国工厂”违和感
- 内外物流链条仍然能被直观理解为同一产业链的一部分
- 底层物流逻辑无需拆成两套独立模拟系统
- UI、放置、预览与交互在内外保持一致体验
- 内外工况转换在玩法上有意义，而不是单纯视觉遮羞

## 16. 非目标

以下内容不属于本阶段目标：

- 重写现有物流核心模拟
- 彻底移除当前移动工厂内部建造玩法
- 把内部退化成纯 UI 抽象图
- 让内部空间与世界建筑完全一比一写实等比例
- 一次性完成所有资源的完整工况体系
- 一次性完成所有移动工厂 profile 的完整重做

## 17. 风险与注意事项

### 17.1 主要风险

- 若建筑目录与货物工况只做表面换皮，视觉冲突仍会存在
- 若转换层做得过轻，玩家会看不懂内外为何不能直接互通
- 若内部空间仍沿用“地面摆小型外部建筑”的视觉策略，主角进入时仍会出戏
- 若目录分流做得太死，可能导致内容维护成本上升

### 17.2 控制策略

- 明确区分“功能语义”与“具体实现”
- 先做最小闭环，而不是一次性铺满全部建筑
- 优先完成最关键的 1 条示范工业链
- 美术设计必须与数据解耦策略同步推进

## 18. 后续建议

后续可按以下顺序推进设计细化：

1. 明确 `World` 与 `Interior` 建筑目录边界  
2. 明确 `ResourceKind + CargoForm` 数据结构  
3. 明确 `Unpacker / Packer / Buffer` 的玩法角色  
4. 明确内部可行走空间的人行层与物流层结构  
5. 以一条示范产线验证视觉与玩法是否真正统一

## 19. 最终定义

本项目中：

- 固定工厂与移动工厂不是两套不同的工厂游戏
- 它们是同一工业系统中的两种工业标准
- 世界侧代表“野外/固定设施标准”
- 内部侧代表“舰载/车载模块标准”
- 两者共享底层物流语言
- 两者通过工况转换建筑完成资源流转
- 两者在视觉上应形成明确亲缘关系，而非缩放关系

**最终设计口径：**

# 移动工厂封包模块舱内输入端口事故复盘与经验教训

## 1. 文档目的

本文记录“移动工厂里的封包模块在舱内无法正常输入、表现与实际不一致、修复过程反复拉扯”的完整事故复盘。目标不是只写“最后怎么修好”，而是把以下问题讲清楚：

- 这个问题到底是什么。
- 为什么前面看起来像修了很多次，但用户体验上始终没有真正解决。
- 为什么同一个模块会同时出现“喂不进去”“隔空能输入”“预览箭头错位”“地图校验不一致”等互相看似矛盾的症状。
- 这个问题暴露了项目底层设计中的哪些结构性风险。
- 后续遇到同类 bug 时，应该优先怎么排查，避免再走弯路。

## 2. 事件摘要

本次事故表面上是“封包模块在工厂内部收不到东西”，但实际根因不是单一 bug，而是同一个建筑在多个系统中的定义长期不一致。

最终定性如下：

- 用户预期的封包模块是“真正的 2x2 主体，舱内输入贴主体边缘”。
- 运行时长期生效的却是“2x2 主体 + 外挂共享输入口/输出口”的旧转换器模型。
- 物流收发、预览箭头、地图校验、结构视觉、边界接口等多套系统分别引用了不同层次的假设。
- 早期修复主要解决了“旧共享端口模型下的解析失败”，但没有第一时间把结构模型本身改成用户要求的样子。
- 因此，前期每次修复都只解决了症状链上的一段，用户看到的整体行为仍然是错的。

一句话总结：

> 这是一次“结构模型定义漂移”引发的系统性事故，而不是一个孤立的端口判定 bug。

## 3. 用户侧主要症状

本次事故在用户视角上先后表现为：

- 封包模块没有可用“配方”，导致无法封包。
- 增加配方后，仍然无法从舱内供料轨或机械臂把小件送进封包模块。
- 某些情况下又能送进去，但像是“隔着一个空格”被吸进去。
- 预览时显示了多个端口，但实际只有一个方向或一个端口真正生效。
- 有时端口箭头消失。
- 有时箭头显示出来了，但在格子中间，无法表达真实连接方向。
- 解包/封包模块在内部摆放时，预览与实际物流方向不一致。
- 世界侧输出口缓存了货，但没有继续吐到世界传送带。
- 地图验证、预览提示、真实运行状态对同一模块给出不同结论。

这些症状看起来很多，甚至互相矛盾，但它们都围绕同一个核心问题展开：系统内部对“封包模块到底占哪些格、从哪里收、往哪里吐、谁拥有端口、箭头该画在哪里”没有统一真相源。

## 4. 为什么这个问题会显得特别难修

这个问题难，不是因为算法复杂，而是因为它跨了太多层：

1. 配方层
   封包/解包有模板、自动模板、手选模板、一对一模板约束。

2. 结构层
   封包模块不是单格建筑，而是多格建筑。

3. 物流层
   belt、inserter、storage、transfer buffer、boundary port 的收发规则不一样。

4. 标准转换层
   舱内小件和世界大包属于不同 cargo standard。

5. 表现层
   预览箭头、输入槽表现、机体视觉、世界/舱内显示所有权都可能误导判断。

6. 校验层
   地图校验与 focused validation 自己也维护一套连接推断逻辑。

7. 兼容层
   packer/unpacker 作为“特殊转换器”被多个地方单独 special-case 过。

换句话说，这个 bug 不是某个函数写错，而是多个系统各自带着不同历史模型在跑。只修其中一层时，其它层继续沿用旧假设，于是用户仍然会感觉“还是没修”。

## 5. 关键概念：项目里其实同时存在三种“格子”

本次事故要真正讲清楚，必须先区分三个经常被混在一起的概念。

### 5.1 建筑占格 `occupied cells`

这是建筑实体真正占住的格子，也决定了摆放冲突、场景中的空间位置和主体轮廓。

### 5.2 输入/输出口格 `input/output cells`

这是物流系统认为“这个建筑从哪些格子接货、向哪些格子出货”的端口定义。

### 5.3 有效来源格 `effective source cell`

这是接收方真正拿来校验“这个物品是不是从合法入口来的”的来源格。

在单格建筑上，这三个概念通常足够接近，所以问题不明显。  
但在多格转换建筑上，这三个概念一旦没有被明确分离，就会出现：

- 占格看起来没问题。
- 端口逻辑在另一组格子上。
- 发送方以为自己对准了。
- 接收方校验时却认为来源非法。

这正是封包模块事故的核心。

## 6. 真正的根因

### 6.1 根因一：封包模块长期运行在“外挂共享口”模型上

运行时，`CargoPacker` 在全局 footprint 解析里并没有按照“用户看到的 2x2 主体输入”来工作，而是被硬编码回一个旧的重载转换器 footprint。

旧模型的本质是：

- 主体占 2x2。
- 输入不在主体边缘，而在主体左侧外挂一列共享输入格。
- 输出不在主体边缘，而在主体右侧外挂一列共享输出格。

这直接导致两个后果：

- 从视觉直觉看，它不像一个真正的 2x2 设备，而像“主体旁边还拖着一排 invisible interaction cells”。
- 用户说的“供料轨可以隔着一个空格给封包组件输入东西”其实不是瞬移，而是系统真的把那个“空格”定义成封包模块的输入口。

### 6.2 根因二：物流收发系统把“端口格”和“有效来源格”耦合得太死

默认接收逻辑长期假设：

- 接收方的 `CanReceiveFrom(sourceCell)` 应该直接拿发送方传进来的来源格与 `GetInputCells()` 对比。

旧共享口模型还能勉强工作，是因为：

- 发送方瞄准的是外挂输入格。
- 接收方的合法输入格刚好也是那个外挂输入格。

但一旦把 packer 改成“主体边缘输入”，新的输入格就落在 packer 自己的主体占格上。  
这时如果发送逻辑还把 belt/inserter 的 anchor 格当成来源格，那么：

- 发送方明明是贴着 packer 发货。
- 接收方却会认为来源格不是自己的合法输入格。
- 结果表现就是“看起来对上了，但死活不收”。

这就是为什么本次最终修复不能只改 footprint，还必须同时修正 effective source 解析。

### 6.3 根因三：预览层对 packer/unpacker 有长期 special-case

预览系统长期对 `CargoPacker` 和 `CargoUnpacker` 做了单独分支处理：

- 不是按真实 `GetInputCells()` / `GetOutputCells()` 全量显示。
- 而是折叠成一个主输入口、一个主输出口。

这带来的问题是：

- 真实物流上可能有两条输入/输出口语义。
- 预览却只画一个。
- 用户看见的端口提示不等于实际收发模型。

更糟的是，当输入口本身落在主体占格里时，原有箭头方向计算默认假设“端口格应该在主体外部且与主体正交相邻”。  
一旦这个假设失效，箭头就会出现：

- 消失。
- 方向错误。
- 漂在格子中间。

因此，前期即使逻辑层已经部分修复，预览仍然会持续给出错误暗示。

### 6.4 根因四：地图校验层维护了一套独立且落后的连通推断

地图校验与 focused validation 并不是直接复用运行时的完整物流解析，而是维护了自己的“谁给谁供货、谁接谁输出”的推断规则。

旧规则长期默认：

- 输入格上站着谁，谁就是 provider。
- 输出格上站着谁，谁就是 receiver。

这对简单单格建筑或外挂端口模型还凑合。  
但对“主体边缘输入”的 packer 来说：

- 输入格上站着的就是 packer 自己。
- 真正上游 provider 在它相邻格。

于是就会出现：

- 运行时能动。
- 校验却说没连上。

或者相反：

- 预览看着能连。
- 校验也说连了。
- 运行时实际由于 source cell 不对而拒收。

### 6.5 根因五：表现层修复先于模型层修复，导致“看起来更像对了”，但本质没变

前期有一段修复主要是为了把 packer 的输入槽、socket、lane 更清楚地画出来。  
这个动作本身不是错的，它提升了可视化信息量。  
但如果底层 footprint 仍然是旧共享口模型，那么这些视觉增强只是在“更清楚地描绘错误模型”，而不是“改正模型本身”。

从用户视角看，这就会表现为：

- 我不是看不懂。
- 我是这东西本来就不该长这样。

这是本次修复过程中最重要的认知偏差之一。

## 7. 为什么前面的修复会显得像在打补丁

本次事故之所以拖得长，主要有四个原因。

### 7.1 第一阶段修的是“旧模型里的真实 bug”，不是“用户要的新模型”

早期修复解决的是共享输入口解析失败：

- 物品目标格被其它结构占住时，系统只认占格上的结构，不继续回退到真正拥有该端口的转换器。

这个 bug 是真实存在的，也确实会导致“belt/inserter 对准封包模块时不收货”。  
但是，它解决的是：

- “旧共享端口模型下，本来应该能收，但解析失败”。

它没有解决的是：

- “我根本就不要共享端口模型”。

因此，从用户体验看，问题仍然没有被真正修掉。

### 7.2 第二阶段被表现层噪声带偏

当用户开始持续反馈：

- 箭头没了。
- 箭头在格子中间。
- 预览看起来还是不对。

这会自然诱导排查者去继续修预览和视觉。  
但这些只是“旧模型继续外显”的结果，而不是主要根因。  
如果没有先冻结住真正的结构模型，继续修表现只会把问题描述得更复杂。

### 7.3 第三阶段没有第一时间把 `GetFootprint(...)` 视为总真相源

这个项目里“结构定义”分散在多个地方：

- 结构 definition 自己带 footprint。
- 特定 kind 又在工厂统一入口里被改写 footprint。
- 视觉层还会根据 footprint 再生成样子。
- 预览、校验也各自调用 footprint。

如果排查时没有第一时间追到全局的 `GetFootprint(...)`，就很容易出现：

- 某个类看起来是 `CompactConversionFootprint`。
- 运行时却被工厂入口强制改成 `HeavyConversionFootprint`。

这会让局部阅读代码得出错误结论。

### 7.4 第四阶段是我的判断失误

这一点要明确记录。

我在修复过程中有两个关键失误：

1. 过早把问题理解为“配置/表现没有对齐”，而不是“结构模型本身错误”。
2. 先优先兼容了旧共享端口行为，而没有立即把 `CargoPacker` 改成用户明确要求的 2x2 主体输入模型。

这两个失误不会让代码更坏，但会显著拉长修复路径，也会让用户合理地产生“你根本没改到点上”的判断。

## 8. 最终修复为什么有效

最终有效的修复不是某一个 patch，而是这四个动作必须一起成立。

### 8.1 动作一：把 `CargoPacker` 的运行时 footprint 改成真正的 2x2 主体输入模型

这一步消除了“外挂共享输入格”这层根源。

结果是：

- packer 真正按 2x2 主体来占格和定义舱内输入。
- “隔着一个空格喂进去”的现象消失。
- 结构模型终于和用户预期一致。

### 8.2 动作二：让 direct receiver 命中主体输入格时，effective source 能正确归一化

这一步消除了“明明贴着主体发送，但接收方仍然认为来源非法”的问题。

结果是：

- belt / inserter 正对 packer 主体边缘时，接收方会把目标输入格当成有效来源口。
- 不是只有共享口回退路径才可收货。
- 主体边缘输入从视觉语义变成真实可运行语义。

### 8.3 动作三：预览不再继续把 packer 折叠成旧模型单口语义

这一步消除了“预览继续给错暗示”的问题。

结果是：

- packer 的输入提示回到真实 input cells。
- 当端口落在主体占格时，箭头方向仍然能正确推断。
- 用户看到的摆放提示不再持续误导下一轮排查。

### 8.4 动作四：地图校验从“查占格 occupant”改成“查真正拥有端口的邻接结构”

这一步消除了“运行时能动但校验说断开”的问题。

结果是：

- focused validation 与运行时连通判断更接近。
- 后续排查不会再被旧校验逻辑误导。

## 9. 本次事故暴露出的底层设计问题

本节不讨论要怎么改代码，只讨论底层设计有哪些长期风险。

### 9.1 缺少单一真相源

项目里缺少一个真正统一的“结构交互语义模型”。  
目前至少有以下几套并行事实：

- 建筑类自己的 footprint。
- 工厂入口重写后的 footprint。
- 物流拓扑对输入/输出的解释。
- 预览系统对端口的解释。
- 地图校验对上下游的解释。
- 表现层对输入槽/输出槽的视觉表达。

这意味着任意一个建筑的交互语义都可能在多处漂移。

### 9.2 把“端口位置”“占格”“方向校验”“收发许可”耦合得过深

当前设计并没有把以下几个概念严格拆开：

- 建筑占格在哪里。
- 端口语义在哪里。
- 来源合法性怎么判定。
- 发送目标如何解析。

结果就是一旦建筑从“外挂端口”变成“主体边缘端口”，系统中大量默认假设会同时失效。

### 9.3 特殊建筑的 special-case 过多

packer / unpacker / input port / output port / transfer buffer 这些结构在很多模块里都有单独分支。  
只要某个建筑需要在多个系统中被 special-case，就意味着：

- 新改动必须跨越多个文件同步修改。
- 少改一处就会出现“局部对、整体错”。
- 排查时很难确定谁才是主定义。

### 9.4 运行时逻辑与校验逻辑没有强约束同源

地图校验如果长期维护独立拓扑推断，而不是严格复用运行时同一组交互语义，那么：

- 它迟早会滞后。
- 一旦滞后，就会反过来成为排查噪声。

这次事故已经证明，这个风险不是理论风险，而是现实成本。

### 9.5 表现层可误导性过强

当前系统允许以下情况同时存在：

- 结构视觉画得很合理。
- 预览箭头也能出现。
- 但底层实际仍然不是用户理解的交互模型。

这意味着表现层不是“辅助理解”，而可能成为“假证据”。  
一旦进入这种状态，用户的反馈会越来越情绪化，因为从直觉看确实像“讲不通”。

## 10. 经验教训

### 10.1 先冻结最小场景，再谈修复

遇到这类物流 bug，必须先把场景冻结成：

- 一个来源结构。
- 一个目标结构。
- 一个明确朝向。
- 一个明确 cargo standard。
- 一条最短 handoff 路径。

否则不同层的问题会互相叠加，永远修不干净。

### 10.2 先问“模型是否正确”，不要先问“表现是否清楚”

如果用户说：

- “这玩意本质上就不是 2x2”
- “怎么能隔着空格输入”
- “端口明明不该在那里”

那优先级最高的问题不是“把现有模型画清楚”，而是先确认这个模型是不是错的。

### 10.3 对多格建筑，必须先追全局 footprint

所有多格物流 bug，第一件事都应该是：

- 看运行时真正生效的 footprint 在哪。
- 不要只看结构类里的默认 definition。

### 10.4 预览与校验只能在逻辑模型稳定后再信任

如果逻辑模型未稳定，预览和校验最多只能作为线索，不能作为事实。

### 10.5 兼容旧模型的修复，不应冒充终态修复

如果当前修的是“旧共享口模型下的兼容问题”，必须明确说明：

- 这只是过渡修复。
- 不等于满足“真正 2x2 主体输入”的目标。

否则沟通上会直接失真。

## 11. 后续遇到类似问题时的排查顺序

建议严格按下面顺序排查：

1. 冻结最小复现场景。
2. 打印或确认结构的真实 `occupied cells`。
3. 打印或确认结构的真实 `input cells` 与 `output cells`。
4. 检查发送方实际瞄准的 target cell。
5. 检查接收方实际校验的 effective source cell。
6. 检查结构自身 `CanReceiveFrom` / `CanAcceptItem` 的特殊逻辑。
7. 再看预览箭头是否与真实 input/output 对齐。
8. 最后再看地图校验是否同步。

如果这个顺序被打乱，极容易进入“表现先修、逻辑后补、每次只好一点点”的状态。

## 12. 结论

这次事故说明，移动工厂里的封包模块问题从来不是单点失误，而是一个长期被系统容忍的定义漂移问题：

- 用户心中的结构模型与运行时结构模型不一致。
- 运行时结构模型与预览不一致。
- 预览与校验不一致。
- 兼容旧模型的补丁一度被误认为能解决用户真正诉求。

最终真正起作用的，不是“再补一个端口 if”，而是把模型、收发语义、预览和校验重新拉回一致。

最重要的经验教训是：

> 对多格物流建筑，必须先统一“结构占格、端口语义、来源判定、预览表达、校验推断”这五件事的真相源。只要这五件事仍然分散在不同系统里独立演化，同类事故还会继续发生。

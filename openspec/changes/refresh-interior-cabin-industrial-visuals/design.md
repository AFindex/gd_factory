## Context

`define-dual-industrial-standards` 已经把移动工厂舱内的“规则身份”从世界侧设施中拆了出来，但当前项目的视觉层仍然主要依赖世界结构的程序化体块、世界货物的默认轮廓，以及少量站点相关的文案替换。这意味着系统在逻辑上已经接受“舱内是另一套工业标准”，但在可见层上仍然像是“把世界设施缩小后塞进船壳里”。

这次改动是一个典型的跨模块表现重构：会同时影响结构可视化、货物可视化、边界接口、编辑器预览和回归验证，但不应该反向污染底层物流模拟、站点拓扑或配方逻辑。约束很明确：

- 继续复用既有的 `FactoryStructure` / `FactoryTransportVisualFactory` / `MobileFactoryDemo` 体系，不拆物流模拟。
- 舱内表现必须服务于“维护空间 + 模块层 + 物流层”的阅读方式，而不是只做颜色或缩放上的区分。
- 需要允许不同舱内对象共享一套家族化视觉语言，避免每个结构各自手写一套不一致的体块。
- 必须保留世界与舱内在功能上的亲缘关系，但不能继续共享同一外部设施轮廓。

## Goals / Non-Goals

**Goals:**
- 建立一套可复用的舱内工业视觉语言，让内部物流件、模块舱、缓存柜、硬点和接口具备一致的“舰载/车载模块”家族特征。
- 让内部专用建筑、边界接口和 `InteriorFeed` 货物在轮廓与载体上与世界侧形成明确差异，而不依赖单纯缩放或重命名。
- 让内部编辑器预览、标签和边界提示和新的舱内模型语言一致，帮助玩家读懂维护层与物流层的关系。
- 为演示地图、headless validation 和 smoke 增加足够的回归保护，避免后续再次把世界侧表现误复用进舱内。

**Non-Goals:**
- 不重写 belt / splitter / merger / placement / topology 等底层物流语法。
- 不重新定义世界站点的整体视觉标准；本次重点是舱内专用对象与舱内版本对象。
- 不要求第一阶段引入完整外部 DCC 资产流水线；程序化模型、局部共享 builder 或少量 scene 资源都可以接受。
- 不在这次 change 中重新平衡内部面积、配方节奏或移动工厂生命周期玩法。

## Decisions

### 采用“舱内视觉角色矩阵”统一不同对象的表现语言

为舱内表现建立一组明确的视觉角色，例如：`EmbeddedLogistics`, `ServiceModule`, `BufferCabinet`, `PowerNode`, `Hardpoint`, `HullInterface`。结构和货物不是各自随意做“更像内部”的造型，而是先落到某个角色，再从角色继承共同的壳体、面板、槽道、检修面和信号灯语言。

这样做的原因是，本次范围覆盖建筑、货物、边界接口和预览。如果继续逐个结构手工微调，短期能出效果，但长期会重新回到风格漂移和重复劳动。相比“每个类单独写一版内饰 builder”，角色矩阵更适合形成统一舱内语言。相比“整套切到外部模型资产”，这个做法也更适合当前项目仍以程序化和轻量资产为主的现状。

### 结构表现按“站点上下文 + 视觉角色”分流，而不是复制 simulation 类

结构逻辑类继续保留统一 simulation 身份，但其 visual-profile 解析需要允许读取站点上下文，并在内部站点选择舱内版本 builder、scene 或视觉装配辅助函数。对于兼有世界版和舱内版的结构，差异应体现在 visual profile / presentation helper，而不是复制出两套行为类。

相比“为内部再派生一套结构类”，这种方式能避免 simulation、配方和交互逻辑分叉。相比“统一结构只加一些底座和配色”，它又能让内部版本真正换成模块舱、抽屉仓、槽道和炮井式轮廓。

### 舱内货物通过“载体家族”而不是单个裸物体来表达

`InteriorFeed` 及其衍生载体不再以世界矿块、板材或箱装轮廓的缩小版呈现，而是映射到舱内载体家族，例如供料卡匣、密封料盒、抽屉仓、弹仓盒、维护匣等。资源语义通过颜色、标记和局部轮廓识别延续，但首先让玩家感知到“这是舱内供料载体”。

相比“只改 tint 或 scale”，载体家族能真正回答为什么内外同种资源长得不同。相比“每个资源都做完全独立资产”，载体家族又能控制资产和 builder 数量，让同类物品共享底座和包装结构。

### 边界接口被视为“壳体接口层”，而不是舱内普通物流件

`InputPort`、`OutputPort`、采矿接入口以及相关预览、部署连接件，应在视觉上被定义为壳体标准接口：包含舱壁法兰、密封 collar、维护 hatch、导向槽或伸出式硬连接，而不是把世界侧 port 或 belt 接头缩小后放进内部网格。

相比把边界接口仅当作普通结构渲染，这能更好地把“标准转换”表现成空间上真实存在的工业边界。相比新增一整套专用系统，也仍然可以复用既有 attachment 数据和部署逻辑。

### 用可视化回归验证保护“舱内不再复用世界轮廓”这一目标

这次改动的风险不在逻辑崩溃，而在后续回归时又悄悄复用了世界模型或默认方块。为此，headless validation 和 smoke 需要覆盖至少三件事：舱内专用货物 descriptor/载体家族是否正确、关键舱内结构是否走了舱内 presentation 分支、边界接口与编辑预览是否暴露舱内专用标签或轮廓。

相比只靠肉眼检查 demo，这类断言更适合长期保护“内部不是缩小版世界”的方向。

## Risks / Trade-offs

- [舱内视觉角色定义过粗，最后仍然只是统一底座 + 换色] → Mitigation: 在 spec 中明确要求物流槽道、模块舱、缓存柜、硬点和壳体接口至少形成可辨认的轮廓差异，而不是只有配色差异。
- [结构 visual profile 分流后，程序化 builder 复杂度上升] → Mitigation: 提炼共享 helper / style catalog，减少每个结构重复拼盒子。
- [货物载体家族数量过多，导致资产维护成本上升] → Mitigation: 先按 cargo form 和功能族做 3-5 个基础 carrier family，而不是每种资源都做独立模型。
- [边界接口、编辑器预览和世界 miniature 三处表现不一致] → Mitigation: 让它们共享相同的命名、颜色和接口角色定义，并在 smoke 中校验关键文本/descriptor。
- [前一 change 的 dual-standard 规则尚未 archive，文档口径容易错位] → Mitigation: 本 change 明确以 dual-standard 已成立为前提，只补“表现层兑现”，并在 specs 中直接引用 industrial-standard / maintenance-space 术语。

## Migration Plan

1. 先建立舱内视觉角色与共享 style helper，明确哪些结构/货物/接口属于哪类舱内家族。
2. 先迁移舱内最关键的对象：供料轨、解包/封包、转运缓冲、内部仓储、内部电力节点、模块舱、炮塔硬点、边界接口。
3. 再迁移 `InteriorFeed` 与相关舱内货物 carrier family，保证物流流动时也能明显区分。
4. 更新内部编辑器预览、标签、说明文案与地图演示，使预览语言和运行时表现一致。
5. 扩展 smoke / validation 断言，最后再补剩余低优先级内部对象。

回滚策略：

- 若某些结构的新舱内 builder 质量不够，可以局部回退到旧 builder，但必须保留新建的视觉角色接口和 specs，不回退总体方向。
- 若部分高保真模型资源暂不可用，可临时使用程序化 cabin builder 占位，只要不重新落回世界设施轮廓即可。

## Open Questions

- 是否需要为同一结构同时支持“程序化舱内版”和“authoring scene 舱内版”的双层 fallback，还是第一阶段统一走程序化 cabin builder？
- 舱内货物是否要区分更多 carrier family，例如普通供料盒与弹药供料盒，还是先以 `InteriorFeed` / `InteriorAmmoFeed` 两级起步？
- 边界接口在世界 miniature 中是否需要保留完整壳体接口轮廓，还是只在内部编辑器与近景厂体上完整显示？

## Context

当前项目已经同时维护静态工厂 demo、聚焦移动工厂 demo，以及共享的移动工厂 HUD / 内部编辑 UI。静态工厂和移动工厂内部都已经切换到真实配方、真实物流与真实供电链，但这也带来了一个明显的调试摩擦点：想验证某条运输线、某个结构详情面板、某个蓝图应用结果，往往必须先搭好供料和供电前置条件。现有 `Producer` 更偏 legacy 占位结构，既没有面向当前工业标准的分类入口，也不适合作为世界与舱内统一的调试工具。

另一方面，移动工厂 HUD 在上一轮“单主面板 + 独立编辑操作面板”的基础上，仍保留了顶部 workspace chrome 与总览 panel 的分离结构，且编辑操作面板顶部还放着一批总览型信息，导致用户在编辑模式里仍要在多个区域来回确认“当前工作区、当前总览、当前建造分类”。

这次设计需要同时覆盖两条横切面：

- 建造系统与结构实现：新增一组真正面向调试的世界建筑 / 舱段模块。
- HUD 与工作区结构：收紧移动工厂 demo 的面板职责，让总览和 tab、建造和编辑操作分别归位。

## Goals / Non-Goals

**Goals:**
- 为世界站点与移动工厂内部站点提供一组零输入、零成本、稳定可复现的调试建筑/舱段。
- 为世界与舱内各补充一个永久运行的测试发电机，确保调试链路不依赖燃料供给。
- 让普通工厂 demo 与移动工厂 demo 都能从建造分类中直接访问这些调试工具，而不改变默认 authored 启动布局。
- 让移动工厂编辑操作面板只承担“建造入口与建造分类”职责，不再重复承载总览信息。
- 将移动工厂顶部 workspace tab 与总览 panel 视觉集成，并为总览 panel 提供右侧折叠/展开按钮。

**Non-Goals:**
- 不把整个工厂系统改成“创意模式”或全局取消资源/电力约束；调试建筑是显式补充，而不是替换正常工业规则。
- 不重做静态工厂 HUD 的布局结构，这次只要求它能暴露新的调试建筑入口。
- 不引入新的复杂调试配置器窗口，例如在建筑详情里动态选择任意物品列表、速率曲线或脚本表达式。
- 不改变移动工厂控制模式、蓝图协议或详情窗口的核心契约，除非实现集成总览 tab 所必需。

## Decisions

### 1. 调试建筑采用“少量固定用途原型”而不是一个万能生产器

本次新增一组显式的 debug build prototypes，而不是扩展现有 `Producer` 或添加一个需要二级配置的“万能调试机”。建议按站点标准提供若干固定用途原型，例如：

- 世界调试供料建筑：覆盖矿物/基础材料、加工材料、战斗补给等常见调试物品簇。
- 舱内调试舱段：提供与世界侧对应的内部物流输入版本，保证内部测试时能直接产出 `InteriorFeed` 兼容货物。
- 永久测试发电机：世界版与舱内版均无需燃料，持续输出稳定电力。

这样做的原因是：

- build palette 中可直接点选，符合当前 demo 的“从建造分类进入工具”心智。
- smoke test 更容易验证固定结构行为，不需要再驱动额外配置 UI。
- site-aware 命名、图标和预览文案可以继续复用现有 `FactoryIndustrialStandards` 体系。

备选方案是保留一个万能生产器，并通过详情面板选择输出物品。这个方案功能最全，但会引入额外 UI、序列化状态与详情交互复杂度，也不符合用户希望“直接在建造分类里补充几个调试用建筑/舱段”的需求，因此不采用。

### 2. 调试生产与测试发电机保持“零成本显式例外”，不污染正常结构规则

调试结构会作为独立原型实现，不共享普通生产机或普通发电机的输入约束：

- 调试供料建筑不消耗原料，也不要求外部输入。
- 永久测试发电机不消耗燃料，也不会因为库存为空而停机。
- 这些结构默认通过名称、分类、文案和视觉强调为“调试专用”，避免与正常 progression 混淆。

同时，默认 authored demo 布局不直接依赖这些结构维持主循环。它们只作为玩家/开发者手动放置时的辅助工具存在。

备选方案是给普通 `Generator` 增加“调试模式”开关，或者给普通机器加一个无成本 override。这个方案会把调试语义渗入通用结构和运行时快照，增加误用风险，因此放弃。

### 3. 调试结构的 palette 集成通过工业标准目录完成，而不是在各 demo 单独硬编码

世界与舱内调试入口统一通过 `FactoryIndustrialStandards.GetBuildCatalog` 暴露，新增专门的调试分类，或在现有分类末尾追加明确标注的 debug 区块。这样：

- 静态工厂 HUD、移动工厂世界模式、内部编辑面板能自动拿到一致的分类顺序。
- site-specific label、preview summary、compatibility 校验保持单一来源。
- 将来若大场景 demo 或其他调试面板复用同一 catalog，不需要重复 hardcode。

备选方案是在 `FactoryHud` 与 `MobileFactoryHud` 内分别手写调试按钮列表。这个方案实现快，但分类会漂移，且后续新增结构时要维护多份 UI 列表，因此不采用。

### 4. 移动工厂编辑操作面板收敛为“建造分类抽屉”，状态摘要回到总览 panel

编辑操作面板将保留：

- 编辑会话入口/关闭入口
- 与建造直接相关的最小操作按钮
- 建造分类与原型按钮列表

编辑操作面板将移除或显著下放：

- 重复的总览说明
- 生命周期/模式/焦点类摘要
- 与 workspace 导航重复的跳转说明

这些摘要信息统一回到移动工厂总览 panel 中展示。这样编辑模式下用户只需要把右侧面板理解为“建造工具抽屉”，而不是第二个总览页。

备选方案是继续保留现有顶部说明，只做文案压缩。这个方案无法真正减少视觉层级，也不能满足“只保留建造分类这些”的目标，因此不采用。

### 5. 顶部 workspace chrome 并入总览 panel 头部，形成总览内嵌 tab 区

当前 `_topChromePanel` 与 `_infoPanel` 分离。新设计中，总览 panel 会拥有自己的 header 区，header 中同时承载：

- “移动工厂总览”标题
- workspace tab chrome
- 右侧折叠/展开按钮

这意味着 workspace tabs 在视觉上成为总览 panel 的一部分，而不是悬浮在上方的独立条带。行为上仍沿用现有 `FactoryWorkspaceChrome`，但挂载位置和布局计算改为由总览 panel 统一管理。

这样可以最大化复用现有 workspace 逻辑，同时消除两层 panel chrome 并排造成的冗余。

备选方案是完全移除独立 chrome 组件、改写一套 tab 控件。这个方案收益不高且回归风险更大，因此不采用。

### 6. 总览 panel 的折叠采用水平滑动状态，而不是直接隐藏节点

总览 panel 新增一个右侧按钮，控制 panel 在“展开”和“收起”之间切换。收起时：

- panel 以动画向左滑出大部分宽度，只保留一条可点击的边缘或按钮宿主。
- workspace 当前选中状态保持不变。
- 编辑视口与世界可见区域自动扩展，占用释放出的屏幕空间。

展开时恢复原宽度，并继续显示上一次 active workspace 内容。此状态只影响 HUD 表现，不改变控制模式和编辑会话状态。

备选方案是简单 `Visible = false`。这个方案会让按钮失去稳定锚点，也容易在状态恢复时丢失上下文，因此不采用。

## Risks / Trade-offs

- [调试建筑过多会污染正常 build palette] → Mitigation: 使用独立“调试”分类或清晰的 debug 命名，并把数量限制在少量高价值原型。
- [无成本发电机和供料机可能被误当作正式玩法结构] → Mitigation: 在名称、描述、颜色与分类中明确标注“调试/测试”，且默认 authored 地图不使用它们维持主循环。
- [总览 panel 与 workspace chrome 重组后，现有布局代码可能出现尺寸回归] → Mitigation: 复用现有 `FactoryWorkspaceChrome` 和 workspace 可见性逻辑，只调整宿主容器与布局计算。
- [总览折叠会与编辑面板动画叠加，导致窄屏拥挤] → Mitigation: 把总览宽度、折叠残留宽度和编辑宽度纳入同一组布局计算，避免各自独立硬编码。
- [共享 `MobileFactoryHud` 可能让大场景 demo 也继承布局变化] → Mitigation: 以共享 HUD 为默认实现，但仅对聚焦 demo 必需的行为写入规格；大场景若沿用共享实现，确保不会破坏其现有 workspace 集合。

## Migration Plan

1. 扩展 build prototype 与结构工厂，加入调试供料建筑/舱段和永久测试发电机。
2. 更新 `FactoryIndustrialStandards`、展示标签、palette 分类与预览文案，让世界/舱内目录都能暴露调试分类。
3. 让静态工厂 HUD 与移动工厂世界/内部编辑面板通过共享 catalog 自动显示新调试条目。
4. 重构 `MobileFactoryHud` 布局：把 workspace chrome 挪入总览 panel 头部，并新增总览折叠状态与布局动画。
5. 精简编辑操作面板，只保留建造分类相关内容，移除重复摘要与导航说明。
6. 更新 smoke / regression 覆盖，验证调试建筑行为、总览折叠与移动工厂编辑面板的新最小结构。

回滚策略：

- 若调试结构引发 catalog 或运行时回归，可先保留结构实现，但暂时从 build catalog 中移除入口。
- 若 HUD 集成造成 workspace 交互问题，可先恢复独立 `_topChromePanel`，保留编辑面板精简与总览折叠逻辑为下一轮继续收口。

## Open Questions

- 总览 panel 的折叠状态是否需要跨场景或存档持久化。当前建议先只保留运行期状态，不写入持久化数据。
- 调试供料建筑的分组数量是否需要进一步细分到“矿物 / 板材 / 中间件 / 弹药”。当前建议优先从少量高频调试分组起步，避免 palette 膨胀。

## Context

当前项目已经有一套可运行的世界工厂系统和一套可运行的移动工厂内部编辑系统，两者在底层上共享大量物流、站点和放置逻辑，但在上层表现上仍然把许多建筑与物品当成同一工业对象的不同缩放版本。随着世界侧建筑体量、移动工厂内部空间、边界 attachment、玩家角色与内部可进入空间共同出现，这种“同语义直接缩放”的做法开始同时破坏三件事：视觉可信度、主角尺度一致性，以及后续内容扩展的稳定性。

当前设计文档已经明确了方向：固定工厂与移动工厂内部应共享底层物流语法，但不应共享同一套具体建筑目录、货物工况与视觉表现。实现上最重要的约束是：不能为了视觉问题重写 belt / splitter / merger / placement / topology 这些已经具备价值的底层；需要优先通过目录分流、工况分层、转换建筑和内部维护空间语义来解决问题。

## Goals / Non-Goals

**Goals:**
- 保留现有世界与内部物流系统的共通底层语法，包括连接、流动、阻塞、预览和交互规则。
- 引入 `World` 与 `Interior` 两类工业站点目录，使“能建什么、能运什么、能加工什么”主要由配置与站点规则决定。
- 将资源语义与货物工况拆开，使同一资源在世界侧与内部侧能够拥有不同运输形态与视觉体量。
- 在内外工业标准之间加入明确的转换层，让 boundary attachment、解包机、封包机与中转缓冲成为工业链的一部分。
- 让移动工厂内部成为主角可进入的真实维护空间，同时避免“主角站在微缩工厂里”的违和感。

**Non-Goals:**
- 不重写现有 belt、splitter、merger、bridge、storage、placement 和 topology 的底层实现。
- 不要求第一阶段一次性把所有现有建筑都拆成世界版与内部版。
- 不将移动工厂内部改成纯 UI 抽象视图。
- 不尝试让移动工厂内部与世界工厂在物理比例上完全写实一致。
- 不在第一阶段解决所有 profile、所有地图和所有视觉资产的最终版本问题。

## Decisions

### 采用“统一物流行为 + 分离工业目录”的两层架构

底层物流行为保留统一实现，上层把具体建筑拆成“行为身份”和“站点目录身份”两个概念。具体来说，belt、splitter、merger、bridge、buffer、port 这些物流件继续共享现有连接与输运语法，但 `WorldBuildCatalog` 与 `InteriorBuildCatalog` 决定每个站点到底暴露哪些建筑、使用哪些 footprint、外观和限制。

这样做的好处是可以保住已经成型的工厂底层，同时允许外部重型传送带与内部供料轨道在视觉、名称、约束和可接受货物上完全不同。相比“继续让内部和外部共用同一种具体建筑定义”，这种做法更能从根上解决比例冲突。相比“内外各做一套独立物流系统”，这种做法的实现成本和回归风险都更低。

### 将资源语义与货物工况解耦，而不是继续缩放同一物体模型

系统中的资源需要拆成两个层面：`ResourceKind` 表示“是什么资源”，`CargoForm` 表示“以什么形态流动”。例如铁矿仍然是铁矿，但世界侧可以是 `WorldBulk` 或 `WorldPacked`，内部侧则是 `InteriorFeed`。世界带体上运送的大件矿物或箱装货，与内部供料槽道中的卡匣、料盒、供料单元，都可以归属于同一个资源语义，但绝不能继续被表现为简单缩放后的同一个裸物体。

相比继续把 `FactoryItemKind` 直接等同于唯一视觉表现，这种分层能让主角尺度、物体体量和工业空间语义同时成立。相比完全拆成互不相干的物品体系，这种做法仍然保留了完整产业链的可理解性。

### 在内外之间引入显式转换层，而不是让 boundary attachment 直接透传同一工况

内外物流标准之间必须有一个明确的边界，因此 `InputPort`、`OutputPort` 以及新增的解包机、封包机和中转缓冲应共同构成转换层。第一阶段推荐最小闭环为：`IntakeUnpacker`、`ExportPacker`、`TransferBuffer`。边界 attachment 的职责从“直接世界交换口”提升为“工业标准边界接口”，允许它参与货物工况切换和站点规则验证。

这样做能让“内外货物为什么长得不一样”成为玩法可见行为，而不是美术上的隐性补丁。相比只靠文案解释或只靠视觉换皮，这种做法更具系统稳定性，也更容易扩展出前线补给、移动精炼、中转装配等玩法。

### 把移动工厂内部定义为“维护空间 + 模块层 + 物流层”，而不是世界工厂的微缩地面

玩家进入移动工厂内部时，看到的必须是主角尺度下的真实工业舱段。为此，内部空间需要明确区分人行维护层、模块设备层和嵌入式物流层。玩家主要行走在猫道、检修平台和维护通路上；机器以舱段模块、嵌入式设备、接口面板和检修口的形式出现；物流则优先隐藏在地板槽道、墙内导轨、母线和供料系统中。

相比继续把内部建筑当成地面上的小型外部建筑摆件，这种做法更能调和主角尺度、内部物流尺寸和整体视觉风格。相比把内部完全做成抽象图，这种做法也保住了“内部是真实空间”的目标。

### 以目录和规则驱动 MVP，而不是一开始全面重构所有现有内容

第一阶段只需要验证一条最小演示链：世界采矿与世界重型物流进入移动工厂，经解包后转成内部供料，在内部模块完成至少一段精炼或制造，再经封包输出回世界接收端。只要这条链路可以在一个 profile、一组内部建筑、三种货物工况和一套转换建筑上跑通，就足够验证这套架构方向。

相比一开始全面替换所有地图、所有内部案例和所有建筑资产，这种做法可以显著降低风险，并让后续内容迁移更有依据。

## Risks / Trade-offs

- [目录分流会引入更多配置资产和规则表] -> Mitigation: 先只定义 `WorldBuildCatalog`、`InteriorBuildCatalog` 两套最小目录，并通过共享行为定义减少重复。
- [资源语义与货物工况解耦会增加物品表示复杂度] -> Mitigation: 第一阶段只引入 `WorldBulk`、`WorldPacked`、`InteriorFeed` 三种工况，不追求一次到位覆盖全部特殊品类。
- [转换层如果过度显式可能让产线显得啰嗦] -> Mitigation: 仅在跨站点标准的边界上要求转换，内部或世界单一标准内继续走统一物流语法。
- [移动工厂内部若仍缺少明确的人行层和物流层区分，主角进入后依然会出戏] -> Mitigation: 将维护空间纳入 capability 约束，要求内部建筑与视觉表现围绕维护空间设计。
- [维持底层共通但上层分化，容易出现“行为相同但配置不一致”的边界 bug] -> Mitigation: 在 specs 和 tasks 中明确站点规则、目录校验和边界 attachment 的验证职责。

## Migration Plan

1. 建立 `World` 与 `Interior` 两类工业站点目录与规则配置入口，先不改底层物流行为。
2. 引入资源语义与货物工况的分层模型，并让物品表现与建筑可接受工况改为读取该分层数据。
3. 实现最小转换层建筑与边界 attachment 规则调整，使世界与内部之间可以通过显式转换建筑完成工况切换。
4. 更新移动工厂内部编辑与内部视觉语义，使其使用内部专用目录并围绕维护空间组织布局与预览。
5. 用一条最小演示产线和一个移动工厂 profile 验证从世界采集到内部加工再到世界输出的闭环。

回滚策略：

- 若目录与工况分层在第一阶段引入过多回归，可保留底层共通逻辑不动，只临时冻结新目录和新工况在 demo 中的启用范围。
- 若内部维护空间方案暂未准备好完整可视表现，可先保留规则层和目录层，延后角色进入内部的完整呈现，但不回退到“世界建筑缩小版”的方向。

## Open Questions

- 第一阶段是否需要让所有现有制造机器都理解货物工况，还是先只覆盖移动工厂演示链路中的少数机器？
- `TransferBuffer` 应该定义为独立建筑，还是作为某些端口/转换建筑的内置行为？
- 内部维护空间是否需要与现有 split-view 编辑器完全绑定，还是允许后续扩展为第一人称/第三人称可进入模式？
- cargo form 是否需要在存档、蓝图和调试输出中单独可见，还是仅在运行时与视觉系统可见？

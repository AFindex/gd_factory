## Context

当前项目已经建立了世界大包与舱内小包的尺度区分，也已经把输入/输出接口、解包舱和封包舱从普通小型节点拉成了更重载的布局。但现阶段的重载链路仍然偏“through-lane”思维：物品主要以 transit 的方式经过接口，接口、桥接位、缓存位和处理位之间没有被建模为显式阶段，因此很难同时满足下面这几件事：

- 世界侧接口与世界传送带需要视觉上连续对接。
- 舱内侧接口需要先缓存一个完整世界大包，再等待解包舱空闲时交货。
- 输出侧需要先由封包舱交出完整大包，再经内缓存、桥接、外缓存平滑送上世界带。
- 世界大包一旦进入接口或转换舱，仍然要保持世界尺度，不能再被缩放成舱内小包尺寸。
- 同一件大包在任一时刻只能有一个真正拥有者，避免重复显示、首帧错模和瞬移感。

这次 change 是一个跨模块变更，覆盖边界附件、转换舱、结构表现、编辑器反馈和 demo 演示，因此需要先用设计文档把状态机、槽位语义和表现分层讲清楚，再进入实现。

约束条件：

- 不重写底层 belt / topology / placement 核心。
- 不新建一整套独立的世界物流模拟。
- 重载接口、解包舱、封包舱仍然运行在现有站点与结构框架内。
- 首个版本优先实现“推荐方案”而不是做所有升级路线。

## Goals / Non-Goals

**Goals:**
- 把输入/输出接口正式定义为“重载装卸机”，而不是普通边界 transport 节点。
- 为输入和输出两种方向都提供对称的双缓存模型：世界侧缓存 `1` 个大包、舱内侧缓存 `1` 个大包、桥接位最多 `1` 个大包。
- 定义连续交接状态机，让大包可以沿“世界带 -> 外缓存 -> 桥接 -> 内缓存 -> 转换舱”以及镜像反向链路平滑移动。
- 明确解包舱和封包舱与接口之间采用握手式交接，而不是直接把大包塞进普通料轨。
- 要求世界大包在接口、桥接、缓存位和处理位内都维持同一世界尺度。
- 让编辑器、详情、提示和 focused demo 能读出当前处于“等待世界来货 / 正在桥接 / 等待解包 / 等待世界接货”等状态。

**Non-Goals:**
- 不把普通 belt、splitter、merger 等轻型物流节点改造成能运输世界大包。
- 不在这次 change 内扩展多件并行桥接、编组装卸或多车道重载接口。
- 不重做所有移动工厂 profile 的外壳模型，仅保证接口与转换链条可读。
- 不在本次 change 中重新平衡所有配方或世界物流吞吐。

## Decisions

### 1. 将重载接口建模为显式的双缓存状态机

每个 `InputPort` / `OutputPort` 都从“普通 transport 结构 + transit 列表”升级为“重载装卸机控制器”，拥有下列显式槽位：

- `OuterBufferedItem`: 世界侧缓存位持有的完整世界大包
- `InnerBufferedItem`: 舱内侧缓存位持有的完整世界大包
- `BridgeTransitItem`: 当前正在桥接中的完整世界大包
- `Phase`: 当前装卸阶段
- `PhaseProgress`: 当前阶段进度

推荐版本约束写死为：

- 世界侧缓存最多 `1`
- 舱内侧缓存最多 `1`
- 桥接位最多 `1`
- 单次只推进 `1` 个完整世界大包

这样做的原因是，用户想要的是“工业装卸感”，而不是继续让大包躺在一个通用 transit list 里。显式槽位能同时解决节拍表达、重复显示和状态反馈问题。

备选方案是继续沿用 `FlowTransportStructure` 的 transit 列表，只通过额外筛选和更多 visual anchors 模拟缓存。这个方案实现上看似更省，但 ownership 和状态表达仍然是隐式的，容易继续出现重复显示和阶段混乱，因此放弃。

### 2. 输入与输出采用镜像状态机，但交接对象不同

输入方向的推荐状态机：

```text
Idle
 -> OuterBuffered
 -> BridgingToInner
 -> InnerBuffered
 -> HandingToUnpacker
 -> Idle / OuterBuffered
```

输出方向的推荐状态机：

```text
Idle
 -> InnerBuffered
 -> BridgingToOuter
 -> OuterBuffered
 -> HandingToWorld
 -> Idle / InnerBuffered
```

关键区别：

- 输入口的下游是 `CargoUnpacker`
- 输出口的上游是 `CargoPacker`
- 输入口从世界站点拉货；输出口向世界站点放货

之所以用镜像模型，是为了让玩家学一次就能理解两边：外缓存、桥、内缓存、交接目标，顺序一致，只是方向相反。

备选方案是把输入和输出做成两套完全不同的系统。这样会让实现和调试复杂度翻倍，而且不利于后续统一升级，因此放弃。

### 3. 转换舱继续一次只处理一个世界大包，但接口前后增加等待位

`CargoUnpacker` 和 `CargoPacker` 继续遵守“处理位一次只持有 1 个世界大包”的规则，不扩成多件并发处理。吞吐提升的空间留给后续版本通过并列舱体或升级件解决。

为了弥补单件处理带来的节拍停顿，接口侧新增缓存位：

- 输入口的舱内缓存位允许在解包舱忙碌时先停住一个完整大包。
- 输出口的舱内缓存位允许在世界带暂时无法接货时先停住一个完整大包。

这个决策把“处理位”和“等待位”解耦，既保留了单件重载处理的可信度，也不会让整个链条因为解包/封包时长而显得笨重。

备选方案是把缓存直接塞回解包/封包舱内部 buffer。这样虽然数据结构少一点，但玩家无法读出哪个大包是在“等处理”、哪个是在“处理中”，视觉也会继续拥挤，因此放弃。

### 4. 重载链路使用连续路径和阶段性交接，不使用瞬移切换

所有大包在下面这些段之间都走连续路径：

- 世界带末端到世界侧托盘
- 世界侧托盘到桥接位
- 桥接位到舱内侧托盘
- 舱内侧托盘到解包/封包舱处理位
- 输出方向镜像返回到世界带

同一件大包在路径上的显示始终使用同一世界尺度和同一 cargo identity，只在 ownership 转移时切换所属槽位，不允许“在 A 隐藏、在 B 新生成一个缩放版本”。

这样做可以最大限度满足“无缝”的观感，也能从根上减少首次显示错模型和多处同时出现同一件货物的问题。

备选方案是保留逻辑分段，但每段间做淡入淡出切换。这个方案只能补视觉，无法让玩家真正相信是同一件大包在被处理，因此放弃。

### 5. 解包/封包舱的淡出发生在处理完成节点，而不是接口中途

输入方向中，接口只负责把完整世界大包送到解包舱。真正的退场发生在解包舱内部：

- 大包进入处理位后保持完整模型
- 夹持、定位、扫描、拆壳等机械动作开始
- 在“完成解包”的明确节点之后才做壳体淡出或拆解退场
- 随后开始吐出舱内小包

输出方向镜像：

- 小包在封包舱内部累积
- 满足模板后生成完整世界大包
- 完整大包离开封包舱，进入输出接口内缓存

这个决策能避免接口过于像魔法门，也能让解包/封包舱真正成为跨标准转换层的视觉核心。

备选方案是在接口中部就完成 fade，解包舱只负责产出小包。这个方案会削弱转换舱存在必要性，因此放弃。

### 6. 保留现有站点与结构框架，但在重载接口层增加专用状态机和握手

实现上不推翻现有结构系统，而是在当前框架内增加一层“重载状态控制”：

- 接口结构负责缓存、桥接和与世界站点的收放货
- 转换舱结构负责处理位、节拍和处理完成后的交付
- 两者之间通过显式 `ready / can_accept / take / release` 握手推进
- 轻型 belt 只继续承接 `InteriorFeed` 小包，不再碰完整世界大包

这让 change 仍然可以沿着现有 `FactoryStructure` / `SimulationController` / `FactorySite` 体系实现，而不需要把物流内核整体拆开重做。

备选方案是为大包再做一整套平行物流系统。这样边界更“干净”，但实现量和回归风险都太高，不符合当前阶段目标，因此放弃。

## Risks / Trade-offs

- [重载接口状态机引入新的复杂度] → Mitigation: 把状态机限制为显式的 3 个槽位和少数阶段，避免做成泛化编组系统。
- [接口与转换舱握手可能和现有 `FlowTransportStructure` 行为冲突] → Mitigation: 明确区分“轻型 transit”与“重载 handoff”路径，让大包在接口层脱离普通 belt 语义。
- [双缓存会让玩家误以为接口吞吐已经很高] → Mitigation: 在提示和动画上强调“缓存位”和“处理中”区别，吞吐仍受单件处理节拍限制。
- [如果世界侧带路不畅，输出口可能长时间停在外缓存] → Mitigation: 明确外缓存就是合法等待位，并在详情与编辑器中显示“等待世界接货”状态。
- [视觉连续路径实现不当会造成穿模或遮挡] → Mitigation: 在设计层预留独立锚点和路径定义，必要时在 shell 开口和观察窗上同步调整可视范围。
- [focused demo 和 smoke 容易继续沿用旧 through-lane 预期] → Mitigation: spec 明确要求 demo 演示完整链路，任务中单独列出 smoke / map /提示更新。

## Migration Plan

1. 先补齐 proposal、design、specs，把重载接口的状态机、缓存位和转换舱握手定义清楚。
2. 实现输入/输出接口的显式槽位与阶段机，先保证逻辑 ownership 正确，再接视觉。
3. 把解包/封包舱升级为与接口握手的处理位结构，完成“完整大包进入处理位后再退场”的链路。
4. 补连续路径和缓存位视觉，让世界带、接口、转换舱的运动看起来是一条连续工业流程。
5. 更新编辑器、详情、focused demo、world miniature 与 smoke，使新语义可观察、可验证。

回滚策略：

- 如果连续桥接视觉一时做不完，可先保留双缓存和正确 ownership，再用简化路径补过渡，但不能回到缩放/瞬移模型。
- 如果 focused demo 改动较大影响回归，可临时保留旧案例作为对照，但新的 focused 链路必须存在。

## Open Questions

- 首个版本的外缓存与内缓存是否都需要完整的机械托盘动画，还是先保证位置和状态正确、动画后补？
- world miniature 是否需要完整复现双缓存状态，还是只需保留“有货 / 无货 / 正在桥接”的抽象信号？
- 是否需要为输出口增加“世界侧接货超时”的专门提示，帮助玩家理解为什么大包停在外缓存位不动？

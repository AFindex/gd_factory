## Context

当前移动工厂 demo 的 HUD 结构已经偏离 `factory_demo.tscn` 很多。静态工厂 demo 主要围绕“顶部工作区菜单 + 单个主信息面板 + 独立详情窗口”组织，而移动工厂 demo 现在同时常驻世界信息面板、右侧编辑视口、编辑侧栏、模式按钮和焦点提示，导致以下问题同时出现：

- HUD 层级过深，玩家需要先判断自己正在看“世界摘要、编辑摘要、右侧工具栏、详情窗口”中的哪一块。
- `_controlMode` 只覆盖世界侧控制，而 `_editorOpen` / `_hoveringEditorPane` 又单独控制编辑状态，模式概念没有被 HUD 明确表达。
- 世界信息与编辑工具被拆散到不同区域，编辑模式下很多真正要点按的操作仍然埋在当前 HUD 结构内部，不像 `FactoryDemo` 那样有稳定的主工作面。
- 代码上，`MobileFactoryHud.cs` 和 `MobileFactoryHud.Workspaces.cs` 已经出现大量世界面板 / 编辑面板双份标签与状态同步，继续迭代会越来越难维护。

这次设计的目标不是简单“换个样式”，而是重建移动工厂 demo 的交互分层，让它在 HUD 使用心智上向 `FactoryDemo` 靠拢，同时保留移动工厂特有的世界态、部署态和舱内编辑能力。

## Goals / Non-Goals

**Goals:**
- 让移动工厂 demo 的主 HUD 结构基本参考 `factory_demo.tscn`：顶部工作区保持一致，主信息区回归单主面板思路。
- 把“编辑模式”从当前的侧栏展开行为提升为显式操作上下文，并配套独立的编辑操作面板。
- 保持世界可见、编辑可见、详情窗口可独立存在，但让三者职责分层更清楚。
- 降低模式切换歧义，明确玩家控制、工厂控制、部署预览、观察模式、编辑模式之间的输入归属。
- 为后续实现提供可落地的 UI 组织、状态模型和测试切入点。

**Non-Goals:**
- 不重做 `FactoryDemo` 本体，也不要求这次先把两个 HUD 全部抽象成一个通用类。
- 不改变移动工厂的生产、物流、部署合法性、蓝图能力或详情窗口基础契约。
- 不引入新的美术主题；视觉上继续沿用现有 one-bit / industrial HUD 语言。
- 不把编辑模式做成完全独占的全屏编辑器，也不取消世界侧可见性。

## Decisions

### 1. 采用“共享 HUD 外壳 + 编辑态附加表面”的双层 UI 架构

移动工厂 demo 的 HUD 会重构为两层：

- 基础层：对齐 `FactoryDemo` 的顶部工作区菜单、主信息面板、状态标签和工作区切换逻辑。
- 编辑层：仅在进入编辑模式时出现的编辑视口与独立操作面板。

也就是说，移动工厂 demo 不再默认同时把世界摘要、编辑摘要和编辑工具全部常驻在主 HUD 内。平时只保留类似 `FactoryDemo` 的主面板；当玩家显式进入编辑模式时，再临时展开编辑相关表面。

这样做的原因是，`FactoryDemo` 的使用心智已经很清晰：先看当前工作区，再在单一主面板中完成主要决策。移动工厂 demo 如果继续保留当前“双大面板 + 嵌套侧栏”的方案，会持续强化“我不知道哪块才是主界面”的问题。

备选方案是继续保留现有世界面板和编辑面板双列布局，只做视觉压缩。这个方案实现快，但并没有解决认知负担和状态同步复杂度，因此放弃。

### 2. 将“世界控制模式”和“编辑会话状态”拆成两个正交状态轴

当前移动工厂 demo 已经有 `MobileFactoryControlMode`，但编辑打开与否通过 `_editorOpen` 等 UI 状态旁路表达。新的设计中，交互语义分为两层：

- 世界控制模式：`Player`、`FactoryCommand`、`DeployPreview`、`Observer`
- 编辑会话状态：`Closed`、`Open`

编辑模式不是去吞掉世界模式，而是在世界模式之上临时打开一个编辑会话。这样能得到几个好处：

- 进入编辑前的世界控制归属可以被记住，退出编辑后恢复。
- 不需要把状态枚举膨胀成 `ObserverEditing`、`FactoryCommandEditing` 之类的组合状态。
- HUD 可以同时表达“当前世界是谁在吃输入”与“当前是否处于编辑会话”。

备选方案是把编辑模式直接塞进 `MobileFactoryControlMode`。这个方案短期看简单，长期会让部署预览、观察模式和编辑之间产生大量组合状态，因此不采用。

### 3. 编辑模式引入独立的 `EditorOperationPanel`

进入编辑模式后，界面新增一个独立操作面板，作为舱内编辑的主操作入口。这个面板负责承载：

- 当前内部交互模式：交互 / 建造 / 删除
- 当前选中建造原型、朝向、旋转提示
- 蓝图快捷入口、删除/取消入口、保存/测试快捷动作
- 当前选中结构摘要、端口状态、焦点提示
- 编辑相关的快捷键说明和上下文文案

主 HUD 面板则收敛为高层摘要：当前世界模式、生命周期、部署提示、场景总览、工作区内容。这样编辑操作信息不会再挤进世界摘要区域，玩家也不必在“左边读状态、右边找按钮、顶部再切 workspace”之间来回找。

备选方案是把编辑工具继续塞进 workspace 主面板，视口只当预览。这个方案与用户要求“编辑模式有独立交互面板”不一致，而且会让编辑态缺少稳定工具区，因此放弃。

### 4. 编辑模式保持“世界视图 + 编辑视口 + 操作面板”三表面，但只让操作面板承载工具

编辑模式下保留世界上下文可见性，原因是移动工厂部署、世界连接和编辑内容之间本来就有联动关系。最终布局采用三表面协作：

- 世界视图：继续显示当前移动工厂、部署点、世界提示和整体状态
- 编辑视口：显示舱内布局与建造预览
- 编辑操作面板：显示所有可点按的编辑工具和状态摘要

其中只有编辑操作面板承担“主要工具面”的职责。世界视图不再承担编辑工具说明，编辑视口也不承载过多文字与按钮。这样能把视觉和交互职责切开。

备选方案是继续使用当前“编辑视口 + 内嵌编辑侧栏”并把世界摘要做成另一列。这个方案本质上仍然是旧架构，没有完成主次重构。

### 5. 顶部工作区菜单只负责“宏观工作区”，不再承担所有编辑工具切换

新的工作区菜单继续使用与 `FactoryDemo` 接近的语义，只负责切换命令、蓝图、存档、验证、详情等宏观内容。编辑模式本身是一个显式动作入口，而不是要求玩家先切进某个工作区才能看到全部编辑按钮。

这意味着：

- “编辑”工作区可以保留为说明性或上下文性入口，但不应该承担所有编辑工具本体。
- 真正的建造、删除、旋转、蓝图快速动作应进入 `EditorOperationPanel`。
- 工作区切换不会隐式打开或关闭编辑模式，除非用户明确执行编辑入口或退出动作。

这样可以避免工作区与操作模式互相绑定，让“我在哪个 panel 看信息”和“我现在在操纵什么”成为两个更稳定的问题。

备选方案是把“编辑”做成一个永远打开编辑面板的 workspace。这个方案会让 workspace 语义和 mode 语义缠在一起，用户在蓝图/详情/测试之间切换时也容易误以为自己离开了编辑或进入了编辑，因此不采用。

### 6. 焦点路由从“两块 pane”升级为“世界 / 编辑视口 / 编辑操作面板”三段式

当前规格只强调世界与编辑视口之间的 hover ownership。新设计需要把独立操作面板纳入输入路由：

- 鼠标位于编辑视口：内部建造/删除/选中逻辑生效
- 鼠标位于编辑操作面板：只触发 UI 控件，不驱动世界或编辑放置
- 鼠标位于世界视图：按当前世界控制模式处理世界输入

键盘输入则按“编辑会话优先于世界热键”的原则组织：例如编辑状态下的旋转、删除、取消优先交给编辑会话，退出编辑后再恢复世界层热键。

备选方案是继续只判断“是否悬停编辑 pane”。这个方案无法准确表达新操作面板的输入消费规则，会引发误放置和误触发。

### 7. 结构详情窗口继续独立存在，不并入编辑操作面板

移动工厂内部结构已经沿用共享详情窗口契约，且 `factory-structure-detail-panels` 已经明确其独立可拖拽属性。这次不把详情窗口并进新面板，而是让它与 `EditorOperationPanel` 并存：

- 操作面板负责“当前编辑能做什么”
- 详情窗口负责“当前结构具体是什么状态”

这样能避免把操作面板做成一个新的“大而全控制台”，也能继续保持与静态工厂 demo 的交互一致性。

备选方案是让操作面板直接替代详情窗口。这个方案会让面板职责失控，并破坏已存在的共享详情交互，所以不采用。

## Risks / Trade-offs

- [移动工厂 HUD 与静态工厂 HUD 更接近后，代码仍可能重复] → Mitigation: 第一阶段先对齐交互层级与布局职责，必要时再把共享 label / panel 构造提取到共用 helper。
- [编辑会话与世界模式拆分后，状态同步点会增加] → Mitigation: 明确由 `MobileFactoryDemo` 持有世界模式与编辑会话的真状态，`MobileFactoryHud` 只做投影与事件发出。
- [独立操作面板加入后，小屏幕可能显得拥挤] → Mitigation: 面板宽度固定且信息分级，非关键说明进入折叠区或次级 workspace，不让所有状态常驻。
- [工作区与编辑模式并行，用户可能一开始不理解两者区别] → Mitigation: 在 HUD 文案中持续使用“工作区”和“编辑模式”两种明确措辞，并给编辑入口/退出提供稳定按钮与热键。
- [输入焦点从两段变三段后，误触风险上升] → Mitigation: 为世界视图、编辑视口、操作面板提供清晰的 hover/focus 可视反馈，并补 smoke test 覆盖焦点切换。

## Migration Plan

1. 先重构 `MobileFactoryHud` 的基础布局，使其主信息区回归 `FactoryDemo` 风格的单主面板与顶部工作区菜单。
2. 在 `MobileFactoryDemo` 中显式引入编辑会话状态，并梳理世界控制模式与编辑会话之间的切换关系。
3. 实现独立的 `EditorOperationPanel`，把当前分散在移动工厂 HUD 各区域的编辑工具、提示和状态迁入该面板。
4. 调整编辑模式的输入路由，把鼠标焦点与热键优先级升级为“世界 / 编辑视口 / 操作面板”三段式。
5. 更新 workspace 文案、按钮命名、提示语和 smoke test，使新的交互层级具备稳定回归点。

回滚策略：

- 若新面板集成后出现较大输入回归，可先保留新的世界主 HUD 结构，仅将编辑工具暂时回退到当前侧栏容器中。
- 若 workspace 与编辑会话解耦带来太多状态问题，可先保留显式编辑入口，但在第一阶段仍复用现有 `_editorOpen` 流程承载部分行为。

## Open Questions

- 编辑模式是否继续沿用 `F` 作为唯一入口，还是在主 HUD 中增加更显式的“进入编辑”主按钮？
- 独立操作面板应当固定停靠，还是允许像详情窗口一样可拖动？当前建议第一版固定停靠，先保证稳定性。
- “编辑”是否还保留为 workspace 标签，还是改成更偏说明性质的 workspace，由按钮负责真正进入编辑？当前倾向于保留标签，但把它作为上下文信息入口而不是唯一操作入口。

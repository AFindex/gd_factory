## Context

当前项目的主要 UI 都是通过 C# 在运行时构建，而不是依赖 Godot 编辑器中预先配置好的统一 `Theme` 资源。`DemoLauncher`、`DemoNavigation`、`UiShowcase`、`FactoryHud`、`MobileFactoryHud`、`FactoryWorkspaceChrome`、`FactoryBlueprintPanel`、`FactoryStructureDetailWindow` 和 `FactoryPlayerHud` 都各自生成 `StyleBoxFlat`、颜色、边框和按钮状态，因此同类控件在不同界面中已经出现不同的圆角大小、边框颜色、背景透明度和高亮逻辑。

这次变更是一个跨多个 UI 模块的视觉重构，但用户已经明确要求不要影响原有业务逻辑。也就是说，我们不能借着主题改造去重排输入职责、改写 HUD 与场景控制器的事件边界，或者让库存/蓝图/工作区等流程因为样式替换而改变行为。设计上需要把视觉系统抽离成一层可复用的样式入口，再让现有 UI 逐步接入。

另一个现实约束是，当前很多状态依赖彩色文本或高亮面板区分，例如 build/observer/command 等模式提示、工作区按钮激活态、蓝图工作台提示、详情窗标题区等。one bit 风格会削弱颜色编码，因此需要提前定义新的状态表达方式，避免界面失去辨识度。

## Goals / Non-Goals

**Goals:**
- 为项目建立一个共享的 one bit 基础 UI 风格入口，统一面板、按钮、标题条、输入框、滚动容器、槽位、分隔线和状态标签的视觉表现。
- 将基础 UI 调整为黑白高对比、清晰边框、低圆角或无圆角的视觉语言，并减少柔和彩色装饰。
- 保持现有 UI 树结构、事件绑定、输入映射、场景控制器回调和库存/蓝图/工作区业务流程不变。
- 让 `UiShowcase` 成为 one bit 主题的验证场景，覆盖常见控件状态、布局密度和文本可读性。
- 为运行时关键状态提供不依赖多色编码的表达方式，例如反相、边框加粗、前缀标签、显式文案和网格/条纹分区。

**Non-Goals:**
- 不重写 `FactoryDemo`、`MobileFactoryDemo`、玩家控制器或库存/蓝图领域逻辑。
- 不引入新的交互模式、窗口管理系统、可拖拽停靠体系或 UI 持久化配置。
- 不把所有 UI 都迁移成 Godot 编辑器里维护的 `.tres` 主题资源。
- 不在本次改动中为每个界面新增动画系统、音效反馈或像素美术资源包。

## Decisions

### 1. 引入共享的代码式 one bit 样式 helper，而不是分散修改每个类里的 `StyleBoxFlat`

新增一个集中式 UI 样式入口，例如 `FactoryUiTheme` / `FactoryUiStyles` 一类的 helper，用来提供：

- 黑白/灰阶调色板与文本颜色；
- 面板、窗口、标题条、按钮、输入框、槽位、分隔线的标准 `StyleBoxFlat`；
- 选中、悬停、禁用、警告等状态的统一表现；
- 推荐的边距、分隔、最小控件高度和最大圆角值。

`DemoLauncher`、`DemoNavigation`、`UiShowcase` 以及工厂相关 HUD/窗口不再各自硬编码圆角和颜色，而是从该 helper 请求对应样式。

这样做优于逐文件手改样式常量，因为当前样式定义散落在多个类中，直接局部替换会继续放大不一致，也会让后续维护 one bit 视觉变得困难。相比直接引入一个 Godot `Theme` 资源，代码式 helper 更适合当前项目，因为绝大多数 UI 都是在 C# 中动态创建，迁移成本更低，也更容易保留现有布局代码。

### 2. 用“黑白反相 + 线框层级 + 文案标签”表达状态，而不是继续依赖多色强调

one bit 风格下，主要交互状态将使用这些手段区分：

- 默认态：浅色文字 + 深色底；
- 激活/选中态：反相填充（浅底深字）或加粗边框；
- 悬停态：边框强化、轻度反相、分隔线变化；
- 禁用态：降低对比度并保留边框轮廓；
- 重要模式提示：显式前缀文案，例如 `[BUILD]`、`[OBSERVE]`、`[PLAYER]`，而不是仅靠不同颜色。

必要时可保留极少量灰阶层次，但不再依赖多种彩色 accent 传达核心信息。对于警告/危险类内容，优先通过文案、图标字符和区块层级提示，而不是重新引入彩色主题。

这样做优于“只把现有蓝色/黄色/绿色换成几种灰色”，因为后者仍然沿用颜色承担语义的思路，一旦变成 one bit 就容易丢失辨识度。

### 3. 默认采用零圆角或极低圆角 token，并通过共享 token 控制例外

共享样式层会将大多数面板、按钮、槽位和窗口标题条的圆角改为 `0` 或非常小的值，建立更方正、机械化的观感。只有在极少数需要避免视觉锯齿或保留手感的控件上，才允许使用很小的圆角 token，例如拖拽预览或输入框内层。

这样做优于每个类自己决定“稍微缩小一点圆角”，因为用户明确提出减少圆角使用；如果不由集中 token 约束，很容易退回到各处半圆角并存的局面。

### 4. 保持布局与交互骨架稳定，只替换视觉构件和少量装饰节点

实现时优先复用现有控件层级、`PanelContainer`/`MarginContainer`/`Button`/`Label` 组合、工作区分发逻辑、详情窗拖拽逻辑以及蓝图面板事件绑定，只做以下类型的变更：

- 替换样式创建函数；
- 收敛边距、分隔和字号；
- 删除不符合 one bit 目标的背景球体、柔和彩色装饰和过度渐变；
- 在需要的地方新增分隔线、状态前缀或边框强调。

这样做优于趁机重排整个 HUD，因为本次需求的成功标准是风格升级而非业务重构。视觉变更与交互骨架解耦，才能更可靠地避免行为回归。

### 5. 把 `UiShowcase` 视为主题验收场，而不是仅改 runtime HUD

`UiShowcase` 已经包含多种基础控件和状态，是验证共享样式是否覆盖常见按钮、标签、进度条、日志、树视图和弹窗的最好入口。改造时应让它直接复用同一套样式 helper，并把当前偏科幻彩色的背景、面板和强调态改成 one bit 表达。

这样做优于只改游戏 HUD，因为 showcase 能更早暴露边距、字号、选中态和可读性问题，也能为后续视觉迭代提供稳定的人工回归样本。

## Risks / Trade-offs

- [one bit 风格削弱部分状态辨识度] → 通过反相、边框层级、显式模式前缀和一致的控件状态规则补回辨识度，而不是让颜色单独承担语义。
- [共享样式 helper 落地后仍有局部控件漏改] → 先覆盖 launcher、workspace chrome、核心 HUD、详情窗、蓝图面板和 showcase，再做一次全项目样式定义检索，收口剩余 `CornerRadius*` 与重复 `StyleBoxFlat`。
- [去掉圆角后界面显得过硬，局部阅读压力增大] → 用留白、分隔线、字号层级和块面节奏保持可读性，而不是回退到大面积圆角。
- [视觉替换误伤交互热区或面板尺寸] → 保持现有容器结构和事件绑定不动，仅在必要时微调内外边距，并对工作区切换、库存拖拽和蓝图操作做回归验证。
- [启动器和运行时 HUD 采用不同接入路径，最终仍不一致] → 要求两类界面都通过同一 helper 创建基础样式，并让 `UiShowcase` 验证这些共用 token。

## Migration Plan

1. 新增共享 one bit UI 样式 helper，并定义调色、边框、圆角、分隔和状态 token。
2. 先接入复用度最高的基础控件与外框，包括 `FactoryWorkspaceChrome`、`FactoryBlueprintPanel`、`FactoryStructureDetailWindow`、`FactoryPlayerHud` 以及 HUD 中的通用面板/按钮样式函数。
3. 改造 `FactoryHud`、`MobileFactoryHud` 和 `MobileFactoryHud.Workspaces`，将工作区、状态栏、建造面板和提示区统一到新主题。
4. 改造 `DemoLauncher`、`DemoNavigation` 与 `UiShowcase`，确保项目入口和展示场景也遵循同一风格。
5. 做回归验证，确认 launcher 导航、sandbox HUD、mobile demo、large scenario、玩家背包/热键栏、蓝图工作台和结构详情窗的业务流程保持不变。

如果中途发现 one bit 主题影响可读性或交互稳定性，可以按模块回滚到旧的样式创建函数，因为场景控制器、数据模型和输入逻辑不会在这次改动中发生耦合变更。

## Open Questions

- 是否需要为极少数高风险提示保留一类非彩色之外的例外视觉，例如更粗的双边框或闪烁文本，以替代过去的危险色高亮？
- `UiShowcase` 是要完全切换为 one bit 主题展示，还是保留它的“多主题切换”概念，但新增 one bit 作为默认主题？
- 如果 launcher 与 runtime HUD 在字号密度上冲突，是否允许保留两组共享 token（入口页 token 与运行时 token），只要求它们遵循同一 one bit 视觉语言？

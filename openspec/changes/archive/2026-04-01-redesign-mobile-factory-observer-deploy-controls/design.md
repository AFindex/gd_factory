## Context

当前移动工厂 demo 已经具备基础生命周期骨架，但世界侧交互仍然以“鼠标悬停锚点后直接部署”为主，`MobileFactoryInstance` 也只有 `InTransit` / `Deployed` 两种粗粒度状态，[MobileFactoryDemo.cs](/D:/Godot/projs/net-factory/scripts/factory/MobileFactoryDemo.cs) 主要负责悬停预览、部署与回收，而不是操作一台真正能在世界里移动的单位。与此同时，[FactoryCameraRig.cs](/D:/Godot/projs/net-factory/scripts/factory/FactoryCameraRig.cs) 默认始终消费 `camera_pan_*` 的 WASD 输入，这和用户希望“WASD 默认控制移动工厂，只有进入观察模式才控制相机”的交互方向相冲突。

这次变更是一个跨 HUD、输入路由、移动工厂状态机和部署流程的交叉改动。目标不是引入完整 RTS 单位寻路系统，而是在现有 demo 的固定地图和预设物流线上，把移动工厂做成一个拥有清晰命令模式、观察模式、部署预览和自动进场表现的可操作原型。

## Goals / Non-Goals

**Goals:**
- 让移动工厂在世界中拥有可感知的移动态与部署态表现，而不是直接在锚点之间切换。
- 将移动工厂 demo 的 WASD 默认绑定到工厂本体控制，解决当前相机与单位抢输入的问题。
- 提供一个显式的观察模式入口，使玩家能随时切回“像现在一样”的相机观察与平移操作。
- 将部署交互重构为 RTS 风格的落点预览流程，支持预览 footprint、端口与朝向，并在确认后自动移动到位、旋转对齐后部署。
- 保留现有内部编辑分屏，但让它和命令模式/观察模式有一致的输入与 HUD 反馈。
- 为 smoke test 和 HUD 提示建立稳定的交互契约，方便后续实现和回归验证。

**Non-Goals:**
- 不在这次改动中引入导航网格、避障寻路、多单位编队或复杂车辆物理。
- 不重做静态 `FactoryDemo` 的输入方案；本次模式切换与 WASD 重绑定主要针对移动工厂 demo。
- 不扩展成完整“手动驾驶模拟器”，例如加速度模型、漂移、碰撞损伤或自由镜头。
- 不在第一版支持多个移动工厂同时下达部署队列或互相抢位。

## Decisions

### 使用“生命周期状态 + 玩家控制模式”双层模型

`MobileFactoryLifecycleState` 继续描述工厂实体当前是否在世界中部署，但需要扩展到足以表达过渡阶段。设计上将生命周期拆成至少四个状态：
- `InTransit`: 未部署，可接受直接移动或部署指令。
- `AutoDeploying`: 已确认一个合法部署目标，工厂正在自动移动并转向。
- `Deployed`: 已完成部署并持有 footprint / port 预留。
- `Recalling`: 正在从部署态退出并返回移动态表现。

同时在 `MobileFactoryDemo` 侧新增一层玩家控制模式，例如 `FactoryCommand`、`DeployPreview`、`Observer`。这样可以把“工厂当前在做什么”和“玩家当前想控制谁”分开描述，避免 HUD、输入和生命周期互相污染。

备选方案是只用一个大枚举同时表示所有操作态，例如 `Driving`、`ObserverDeployPreview`、`EditorDeploying` 等。这个方案被放弃，因为它会快速膨胀，并让 HUD、输入禁用和部署过渡难以复用。

### WASD 在命令模式下改为控制移动工厂本体

移动工厂 demo 中，默认进入 `FactoryCommand` 模式。此时：
- `W` / `S` 控制工厂沿当前朝向前进/后退。
- `A` / `D` 控制工厂左转/右转。
- 鼠标左键不再直接在世界上部署，而是用于部署预览确认。

这样可以明确满足“移动改为 WASD 移动”和“移动态能移动和转向”的诉求，也能和观察模式形成清晰差异。为了降低实现复杂度，第一版采用运动学驱动即可：在 `_Process` 中按固定线速度与角速度推进 `MobileFactoryInstance` 的世界 transform，而不是引入刚体。

备选方案是让 WASD 继续表示平面四方向平移，再用 `Q/E` 负责转向。这个方案虽然更接近当前相机习惯，但不够像“控制一台载具”，也会弱化“转向到正确方向再部署”的表现价值，因此不作为默认交互。

### 观察模式通过显式按钮和热键切换，并独占相机 WASD

HUD 新增一个清晰的模式切换按钮，例如“进入观察模式 / 返回工厂控制”，同时保留一个单键切换热键以便快速验证。设计上推荐使用 `Tab` 作为观察模式切换热键，按钮负责可发现性，热键负责流畅性。

当控制模式为 `Observer` 时：
- [FactoryCameraRig.cs](/D:/Godot/projs/net-factory/scripts/factory/FactoryCameraRig.cs) 的 `AllowPanInput` 恢复为 `true`，相机重新占用 WASD。
- 移动工厂本体不再响应 WASD 移动/转向。
- 左键主要用于查看部署落点或退出观察后的下一步选择，不直接改变工厂状态。

当控制模式离开 `Observer` 时，相机仍可继续缩放，但平移输入被收回到移动工厂控制。这样用户能在一个明确模式里“像现在一样观察地图”，又不会在默认状态下误移动镜头。

备选方案是用按住空格临时观察。这个方案发现成本低，但会让 WASD 的所有权在长按期间频繁抖动，不适合部署预览、编辑分屏和后续可扩展的 HUD 状态表达。

### 部署采用“两步式 RTS 流程”：进入部署模式、确认合法落点、自动进场完成部署

部署交互设计为：
1. 玩家在命令模式点击 HUD 的“部署”按钮，或使用部署热键进入 `DeployPreview`。
2. 鼠标在地面上移动时显示 footprint、端口格和朝向箭头预览。
3. `Q/E` 在预览阶段旋转部署朝向。
4. 左键确认一个合法落点后，游戏创建“待执行部署命令”，切到 `AutoDeploying`。
5. 工厂自动移动到该锚点的部署中心，随后转向到目标朝向并正式执行部署。

这个流程把“像 RTS 放置建筑一样选择落点”和“工厂本体必须真的移动过去”结合起来，同时保留当前 demo 的鼠标预览优势。第一版不引入真实寻路，而是使用直接 steering：工厂以固定速度朝目标中心推进，在到达位置容差后进入朝向对齐，再在朝向误差进入容差后调用底层部署逻辑。

备选方案是确认落点后立即瞬移到目标并播放短动画。这个方案实现更快，但会削弱“真实的移动态/部署态表现”，与这次变更目标相悖。

### 部署落点的有效性以“预览合法 + 最终部署前再校验”作为契约

在 `DeployPreview` 中沿用现有 `CanDeployAt` 判定显示绿色/红色预览。确认合法落点后，不立即把工厂切到 `Deployed`，而是记录目标锚点和目标朝向。到达并转向完成后，再次调用部署校验并真正预留 footprint / ports。

这样可以避免把“部署中”误算成已部署，也给后续加入更多动态障碍留下空间。如果最终二次校验失败，自动部署应中止并回到 `FactoryCommand`，HUD 给出失败原因。

备选方案是确认时就直接预留目标 cells。当前 demo 只有一台移动工厂，这样做也能工作，但会让“工厂仍在路上时目标已经算已部署”显得不自然，因此本次先不采用。

### 朝向从“固定朝东端口”升级为可旋转部署定义

当前 [MobileFactoryInstance.cs](/D:/Godot/projs/net-factory/scripts/factory/MobileFactoryInstance.cs) 的 footprint 和端口是相对固定偏移，输出桥也默认朝东。这次设计要求把部署朝向纳入移动工厂部署元数据：
- 预览要显示当前朝向对应的 footprint 和端口位置。
- 自动部署完成后，工厂 hull、世界端口 visual、port bridge facing 都要与目标朝向一致。
- HUD 要明确显示当前预览朝向与部署后的连接方向。

为控制复杂度，第一版仍然只支持单个外部端口，但允许该端口随朝向一起旋转。这样足以支撑“自动移动过去然后转向到正确的方向并部署”的需求，又不会立即把内部端口拓扑泛化成多端口系统。

备选方案是继续保持朝东固定端口，仅让 hull 视觉旋转。这个方案会让预览方向和实际连接方向脱节，因此被放弃。

### 内部编辑分屏保持现有结构，但不隐式切换观察模式

`F` 打开内部编辑的能力保留，但编辑分屏不再自动把玩家切到观察模式。设计上把它视作一个与控制模式并列的 UI 上下文：
- 编辑面板展开后，鼠标悬停右侧编辑区时，内部建造输入生效。
- 左侧世界条带仍然展示当前工厂和部署预览。
- 若玩家当前在 `Observer`，则世界条带可继续使用相机 WASD。
- 若玩家当前在 `FactoryCommand` 或 `DeployPreview`，则左侧依旧保留对应的工厂命令/部署语义。

这能最大程度复用当前“按悬停分配鼠标输入”的实现，同时避免出现“打开编辑器时突然抢走 WASD 到相机”的隐性模式跳转。

备选方案是编辑器打开时强制进入观察模式。这个方案更省解释，但会让用户难以理解为什么只是展开右侧面板，左侧 WASD 语义就变了。

## Risks / Trade-offs

- [双层状态机让 demo 代码更复杂] → Mitigation: 把生命周期状态放进 `MobileFactoryInstance`，把控制模式只留在 `MobileFactoryDemo`/HUD，职责分清。
- [WASD 改成单位控制后，老用户会短暂不适应] → Mitigation: 在 HUD 持续显示“当前模式”和切换按钮文案，并保留观察模式按钮作为找回旧控制的明确出口。
- [自动部署没有寻路，可能在复杂地图里显得简化] → Mitigation: 在设计上明确第一版只做直线 steering，用于当前 demo 地图；若后续地图更复杂，再在同一命令层补路径规划。
- [朝向可旋转后，端口/footprint 计算更容易出错] → Mitigation: 将“获取 footprint cells / port cells / bridge facing”统一收口到朝向感知的部署定义函数，并用 smoke test 覆盖四向部署。
- [观察模式、编辑分屏、部署预览三者叠加后可能产生输入冲突] → Mitigation: 定义优先级为“编辑 viewport 鼠标输入 > HUD 按钮 > 当前控制模式对应的世界输入”，并在 HUD 上可视化当前焦点。
- [二次校验失败会让自动部署看起来像命令失效] → Mitigation: 在 HUD 中区分“目标无效，无法开始部署”和“到位时目标已失效，部署已取消”两类反馈。

## Migration Plan

1. 先扩展移动工厂状态与部署元数据，让 `MobileFactoryInstance` 能表达目标朝向、自动部署目标和过渡中的 transform 更新。
2. 重构 `MobileFactoryDemo` 的输入层，把相机输入、工厂控制输入、部署预览输入和编辑器输入拆成清晰的模式判断。
3. 在 HUD 中加入模式按钮、部署按钮、模式标签与更明确的提示文案。
4. 改造部署预览，使其支持朝向旋转、四向 footprint/port 预览和确认后的自动进场。
5. 更新世界 port visual、bridge 绑定和部署/回收流程，使部署朝向成为真实 runtime 数据而非纯视觉状态。
6. 扩展 smoke tests，覆盖默认控制模式、观察模式切换、自动部署成功、部署失败回退、部署态禁止移动、回收后恢复移动等关键路径。

回滚策略：如果实现中发现自动部署或模式切换引发较大回归，可以先保留 HUD 的模式层和 WASD 输入分流，再临时回退到“确认即部署”的旧部署执行方式；现有 dedicated mobile demo 和 static factory demo 的分离可以降低回滚影响面。

## Open Questions

- 观察模式是否需要在 HUD 上以“按钮保持高亮”的方式常驻显示，还是只在状态文案中体现即可？当前设计倾向于按钮高亮，以便快速理解输入归属。
- 自动部署过程中是否允许用户按 `Esc` 取消命令？当前建议允许取消，且取消后返回 `FactoryCommand`，避免演示时被错误命令锁住。
- 在部署态下是否允许原地旋转而不先回收？当前设计不允许，保持“部署态必须先回收，才能再次移动或改变部署朝向”的现有生命周期约束。

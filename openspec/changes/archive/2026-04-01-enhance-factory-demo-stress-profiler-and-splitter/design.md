## Context

当前静态 `FactoryDemo` 的默认布局是由 `CreateStarterLayout()` 直接硬编码的一组小型示例线，足够证明生产器、传送带、分流器、合并器、跨桥、装载器和卸载器可以跑起来，但还不足以形成稳定的高压回归场景。与此同时，`FactoryHud` 仍是一个偏宽的左上角整块面板，信息密度较低；在地图内容变复杂后，它会遮挡更多观察区域，也缺少对性能退化的即时观测。

物流行为上，`SplitterStructure` 继承自 `FlowTransportStructure`，并在 `TryAcceptItem()` 时通过 `TryResolveTargetCell()` 为每个进入的物品预先确定 `TargetCell`。这让分流器在入口端就“锁死”出口选择，一旦某一帧偏好的出口后来被堵住，物品会在分流器出口处整体卡死，即使另一侧出口在那一刻明明可用，也不会改道。要把静态 demo 升级成真正的压力测试场景，这个阻塞行为必须修正，否则复杂拓扑越多，假性死锁越难判断。

## Goals / Non-Goals

**Goals:**
- 把 `factory_demo.tscn` 升级为默认即可运行的高压静态测试场景，包含更丰富的复杂拓扑与长链传送段。
- 将 `FactoryHud` 压缩为更紧凑的观察面板，使主要 HUD 占屏约五分之一，同时保留建造与状态反馈。
- 在 HUD 中持续显示运行期 profiler 信息，至少覆盖 FPS、帧时间，以及对当前 demo 最有价值的热点指标。
- 修复分流器在单侧出口堵塞时错误阻塞另一侧出口的问题。
- 让静态 demo 的 smoke/regression 测试可以验证复杂拓扑、HUD profiler 可见性和分流器局部阻塞回退。

**Non-Goals:**
- 不引入新的建筑原型或更复杂的多 lane 物流模拟。
- 不把静态 demo 重做成独立的第二套大型场景或完整关卡编辑工作流。
- 不尝试复刻 Godot 编辑器级别的全量 profiler；HUD 里的 profiler 以运行期调试可读性为主。
- 不在这次变更中重构整个物流模拟框架，只修复与分流器出口选择直接相关的阻塞问题。

## Decisions

### 用“预设压力段组合”扩展默认工厂，而不是把所有布局继续堆在一串裸 `PlaceStructure()` 调用里

`FactoryDemo` 仍然保持运行时程序化搭建场景，但 `CreateStarterLayout()` 应拆成几组命名明确的 authoring helper，例如直通基线段、分流回汇段、跨桥交叉段、长距离输送段、装卸回环段和局部拥塞观察段。每一段只负责一个拓扑主题，再由主布局方法按地图分区组合。

这样既能保留当前“纯代码搭场景”的低门槛，又能避免后续继续在单个 200 行函数里追加坐标，导致场景既难读也难调。相比改成完全手工 `.tscn` authoring，这种方案更适合快速迭代和 smoke test 断言，因为所有关键位置仍然由代码集中声明。

备选方案是直接新建一个更大的静态测试 scene 并保留当前 `factory_demo.tscn` 不动。这个方案被放弃，因为当前规范里默认主入口就是静态 demo；把回归价值最高的布局直接做成默认入口，更符合“打开项目就能看到高压场景”的目标。

### HUD 采用窄列卡片化布局，并把 profiler 收拢到单独统计块

`FactoryHud` 应从现在的宽标题加按钮矩阵布局，改为更紧凑的窄列面板。核心策略是：
- 继续停留在左上角，避免改变当前输入焦点模型；
- 通过更窄的固定宽度、分组标题、紧凑间距和更短的帮助文案，把主面板控制在大约视口宽度五分之一；
- 将“当前建造 / 当前格子 / 预览 / 朝向 / 吞吐 / profiler / 帮助”整理为若干小卡片或逻辑块，而不是平铺长文案；
- 保留可点击建造按钮，但允许按钮更紧凑，必要时分两行排布。

备选方案是在屏幕底部做横向工具条。这个方案能进一步减少横向遮挡，但会显著改变现有鼠标交互的视觉习惯，而且在展示 profiler 时更容易和底部世界内容重叠，因此不作为首选。

### Profiler 采用“引擎基础指标 + 代码内热点采样”的组合，而不是尝试嵌入完整引擎 profiler

HUD profiler 将分成两类数据：
- 引擎基础指标：FPS、平均帧时间，直接反映当前场景总体流畅度；
- 领域热点指标：例如 simulation tick 耗时、活动物流件数量、场内结构数量、视觉更新耗时或最近一次 topology rebuild 耗时，用于快速判断瓶颈是在物品模拟、场景更新还是布局复杂度。

这些数据更适合通过 `FactoryDemo` / `SimulationController` 内部增加轻量采样结构来维护，再由 `FactoryHud` 统一显示。这样做的好处是数据稳定、可控，而且能直接对应当前项目最关心的热点，而不是暴露一堆对本 demo 价值不高的底层引擎计数。

备选方案是只显示 FPS。这个方案实现最简单，但几乎不能回答“为什么帧率掉了”，不满足用户对“性能热点”的要求。另一备选是尝试接入 Godot 编辑器 profiler 级别的深度数据；这会大幅提高实现复杂度，也不适合 headless smoke 断言，因此不采用。

### 分流器需要把“出口选择”从入站阶段延后到出站阶段，确保局部阻塞不会导致整体假死

当前分流器的问题根源不是连通性判断，而是目标出口选得太早。进入 `FlowTransportStructure` 的物品一旦在 `TryAcceptItem()` 时被写死 `TargetCell`，后续即使另一个出口变空，`TryDispatchItem()` 也只会反复尝试原目标。

这次设计选择把分流器的实际出口判定延后到物品接近离开当前结构时再做。实现上可以保持“左右轮询偏好”这条策略，但需要在 dispatch 前重新检查左右出口的实时可用性：
- 若首选出口可用，则按轮询输出；
- 若首选出口堵塞但另一出口可用，则回退到另一出口；
- 只有当两侧都无法接收时，物品才留在出口缓冲位置等待。

这样既保留了分流器的均衡意图，也避免了单侧短时拥塞把整个分流器锁死。相比“在入站阶段就同时记录一组候选出口”或“让分流器直接维护双队列”，延后决策的改动面更小，也更贴合现有 `FlowTransportStructure` 的生命周期。

### smoke test 增加针对复杂默认布局和局部堵塞恢复的断言

现有 `RunSmokeChecks()` 只验证放置/拆除和默认送达是否大于零，这对高压场景和 splitter bug 都不够。新的 smoke/regression 需要覆盖三类契约：
- 默认场景加载后，存在多个不同拓扑段且吞吐持续增长；
- HUD profiler 区块已经生成并有有效文本；
- 在构造单侧出口堵塞、另一侧畅通的分流器 case 时，可用侧的送达仍会继续增加。

备选方案是完全依赖人工观察。这会让这次改动最重要的价值点无法自动回归，因此不采用。

## Risks / Trade-offs

- [默认静态 demo 变复杂后，初次阅读代码的门槛会上升] -> Mitigation: 使用命名明确的布局 helper，并按地图区域分段组织坐标。
- [HUD 压缩后，信息可能因为文案过长而重新撑大面板] -> Mitigation: 将长文本拆成短标签与数值行，帮助信息改为更短的高频操作摘要。
- [热点统计如果采样过多，反而会增加额外开销] -> Mitigation: 只保留少量关键计时点和聚合计数，不做逐结构深度 profiling。
- [延后分流器选路可能触及 `FlowTransportStructure` 抽象边界] -> Mitigation: 优先通过局部覆写 dispatch/target 解析逻辑实现，不在本次改动里泛化所有运输结构的路由模型。
- [更拥挤的默认布局可能让旧的 smoke 时间窗口不稳定] -> Mitigation: 让断言关注“吞吐是否增长”和“局部堵塞是否恢复”，而不是依赖单个固定送达值。

## Migration Plan

1. 重构 `FactoryDemo` 的默认布局 authoring，把开场线路拆成多个命名压力段并重新组织地图区域。
2. 在 `SimulationController` 和 `FactoryDemo` 中加入轻量性能采样结构，再扩展 `FactoryHud` 的显示层。
3. 调整 `FactoryHud` 布局宽度、分组结构和文案密度，确保默认可视占比明显下降。
4. 修改 `SplitterStructure` 的目标出口决策时机，并验证单侧堵塞时的回退输出。
5. 扩展静态 demo 的 smoke test 与文档说明，覆盖复杂拓扑、profiler 显示和 splitter 阻塞恢复。

回滚策略：如果 profiler 或 HUD 改动带来不稳定 UI 回归，可以先保留复杂默认布局和 splitter 修复，只暂时回退 profiler 块的显示；如果分流器新逻辑影响了其他物流件传递，则可先保留 smoke case 与 layout 改动，再将 splitter 回退到旧实现单独修复。

## Open Questions

- profiler 的“热点”第一版是否只显示 2 到 4 个聚合指标，还是需要按模块排序展示 top-N 文本摘要？当前设计倾向于少量高价值指标。
- 复杂默认布局是否需要在地图里加入轻量地标或分区标线，帮助观察不同压力段？如果纯逻辑布局仍然难读，后续可以补一轮视觉标识。

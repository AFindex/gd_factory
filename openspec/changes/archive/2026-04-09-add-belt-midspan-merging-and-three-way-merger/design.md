## Context

当前物流模拟把大多数运输结构建模为单格 `FlowTransportStructure`，物品传递时通过 `SimulationController.TrySendItem()` 把 `sourceCell -> targetCell` 发送给目标结构，再由目标结构的 `TryAcceptItem()` 和 `CanReceiveFrom()` 决定是否接收。这个模型已经足够支撑直线皮带、分流器、双入口合流器和移动工厂内部运输，但它默认把皮带视为“只有一个输入格、一个输出格”的端对端节点，因此任何指向另一段皮带中段的 feeder 都会被判为未连接。

与此同时，地图 headless 校验和 focused connectivity 诊断并没有直接复用运输结构的全部运行时语义，而是通过 `FactoryMapValidationTopologyHelper` 单独枚举输入格和输出格来分析连通性。如果这次只改运行时结构，不同步更新这些 helper，地图作者会看到“实际能跑，但校验说断开”的假阳性结果。由于同一套运输规则同时服务于静态世界和移动工厂内部，设计需要保证放置、拓扑刷新、运行时调度、校验和 smoke 都围绕同一套中段合流定义工作。

## Goals / Non-Goals

**Goals:**
- 让传送带支持“末端指向另一段传送带占用格”的中段合流，不要求只能连到目标带的输入端。
- 把现有合流器升级为三入口单出口，支持后方、左侧、右侧三路同时汇入。
- 让建造校验、预览提示、拓扑刷新、地图连通性分析和回归测试都识别这两类新物流拓扑。
- 保持世界地图和移动工厂内部共用同一套运输行为，不引入双份规则。

**Non-Goals:**
- 不引入新的专用 junction 建筑类型，也不把皮带系统改造成多格 lane-by-lane 仿真。
- 不在这次变更中重做全部传送带视觉资源，只做能支撑可读性的必要调整。
- 不修改工厂地图文件格式；已有 `structure|Kind|x|y|Facing` 记录方式继续有效。
- 不在这次变更中扩展分流器功能或增加优先级/过滤逻辑。

## Decisions

### 将“中段合流”定义为目标皮带对额外入站来源的接收规则扩展，而不是新建隐藏中继节点

`GridManager.TrySendItem()` 和 `MobileFactorySite.TrySendItem()` 已经允许一个结构把物品发送到另一个结构的占用格，因此最小改动路径是让 `BeltStructure` 在 `CanReceiveFrom()` / `TryResolveTargetCell()` 层面接受更多合法 `sourceCell`，而不是在地图或运行时动态生成一个额外 junction。具体上，目标皮带继续保留唯一的前向输出，但它的合法输入来源从“主输入格”扩展为“除前向输出格以外、能够把物品发送到本格的正交邻格”。

这样做的好处是：
- 世界地图和移动工厂内部都能直接复用现有的单格放置与物品发送接口。
- 中段合流不会要求额外 footprint，也不会破坏现有 map/blueprint 序列化格式。
- 现有 `FlowTransportStructure` 的排队、间距和视觉更新逻辑仍可复用，新的 feeder 只是在同一条前向输出线上串行排队。

备选方案是新增一个“隐式 T 接头”或自动把目标皮带拆成多段复合结构。这个方案会把放置、序列化、删除和聚焦诊断都变复杂，因此不采用。

### 皮带视觉与拓扑优先保留主干朝向，侧向 feeder 作为附加输入而不是重新定义整段朝向

当前 `BeltStructure.RefreshTopology()` 通过 `DetermineInputFacing()` 为皮带挑一个主要输入方向，并据此生成直线或转弯的外观。为了避免一条主干皮带因为出现侧向 feeder 就突然变成弯道，这次设计保留“后方主输入优先、侧向输入次之”的判定顺序：如果皮带后方存在能输出到本格的上游，视觉仍以主干方向绘制；只有没有后向主干时，才让单独的侧向 feeder 决定输入臂朝向。

运行时上，所有被允许的 feeder 都可以把物品送入目标皮带，但视觉第一版只保证主干方向稳定与单侧并入可读，不尝试为每一种多输入组合绘制完全独立的网格贴图。这样能避免为了一个规则变更把整套皮带渲染系统重构掉。

备选方案是为每种“主干 + 左并入 + 右并入”组合建完整视觉集。这个方案可读性更高，但需要更大范围的资源和渲染改造，不适合作为本次变更的前置条件。

### 三入口合流器继续保持单格建筑，只扩展接收面和路径曲线

现有 `MergerStructure` 已经是单格、单出口、双侧输入的运输节点。这次将其升级为三入口时，仍保持单格 footprint 和前向唯一输出，只把 `CanReceiveFrom()` 的合法来源从左右两侧扩展为“后方、左侧、右侧”，并让物品曲线基于实时 `SourceCell` 计算，从而支持三种入站路径共享同一出站前向。

保留单格建筑有三个收益：
- 玩家原有布局里只需换规则，不需要为合流器预留更大的 footprint。
- 放置、移除、蓝图、地图加载和移动工厂内部建造都能继续沿用现有单格建筑流程。
- 三入口合流器能自然补上“皮带中段合流解决局部并线，专用合流器解决高密度三路汇总”的角色分工。

备选方案是把合流器做成 2x1 或十字型 footprint。这个方案会提高可视表达，但也会把 footprint、预览、地图存量内容和多格占用逻辑全部推倒重来，因此不采用。

### 使用共享拓扑 helper 统一建造、校验和 focused diagnostics 对新物流连接的理解

运行时结构是否能接收一个来源，取决于 `CanReceiveFrom()`；而地图校验则通过 `FactoryMapValidationTopologyHelper.GetInputCells()/TryGetInputProvider()` 这套逻辑分析连通性。为了避免“运行时规则”和“校验规则”再次分叉，这次设计会把皮带额外 feeder 的识别和三入口合流器的输入定义抽成共享拓扑 helper，供以下入口统一使用：
- 静态世界与移动工厂内部的放置合法性和预览提示。
- `FactoryMapValidationService` 的 advisory connectivity 与 focus report。
- smoke/regression 中对中段合流和三入口合流器的回归断言。

备选方案是分别在 build preview、地图校验和结构类里写条件分支。这个方案短期更快，但很容易再次出现某个入口忘记同步，导致行为和工具反馈不一致，因此不采用。

### 回归验证以“持续输送”和“无误报”两类断言为主，而不是只检查结构能否放下

这次变更的风险不在于能否建造，而在于建造后是否真的能连续流动，以及地图工具是否会把合法拓扑误判为断开。因此验证重点放在两层：
- 运行时 smoke：构造一个末端 feeder 指向主干皮带中段的案例，以及一个三路输入合流器案例，验证物品会持续送达。
- 工具链验证：对包含这两种拓扑的 authored map 运行 headless validation，确认不会把相关皮带或合流器标成 isolated / disconnected。

备选方案是只做交互式人工验证。这个方案会让最容易回退的工具链问题漏网，因此不采用。

## Risks / Trade-offs

- [一条皮带同时存在后向主干和侧向 feeder 时，视觉不一定完整展示所有输入] -> Mitigation: 视觉优先保持主干稳定，测试重点放在行为正确；如可读性仍不足，再在后续变更里补 richer belt visuals。
- [如果多条 feeder 同时并入同一段皮带，单 lane 串行排队可能降低吞吐或让玩家误解为多 lane] -> Mitigation: 规格中明确它是单输出串行合流，不承诺并行 lane 吞吐。
- [地图校验若继续用旧的输入输出枚举，会对新布局产生大量假阳性] -> Mitigation: 在实现期优先重用共享拓扑 helper，并用 headless smoke 覆盖一张包含新拓扑的 map。
- [三入口合流器改变后，旧场景或说明文案可能仍写成双入口] -> Mitigation: 一并更新原型名称说明、HUD 文案和至少一个 authored/demo 用例。

## Migration Plan

1. 为皮带和合流器补充共享物流拓扑 helper，定义合法输入来源、主干优先级和三入口合流器的接收面。
2. 更新 `BeltStructure`、`MergerStructure` 与必要的 `FlowTransportStructure` 协作点，让中段合流和三入口路径在运行时可持续输送。
3. 调整世界与移动工厂内部的放置校验、预览提示和原型说明，使新拓扑在建造阶段可被理解和允许。
4. 更新地图连通性分析与 focused diagnostics，确保 headless validation 识别新的合法连接关系。
5. 补充 smoke/regression 布局与测试断言，覆盖中段合流输送、三入口合流，以及校验结果不误报。

回滚策略：如果中段合流的运行时实现引发广泛回归，可以先回退皮带额外输入支持，只保留三入口合流器规则和相应文档；如果工具链适配不稳定，可以暂时保留运行时支持并回退新的校验断言，待共享 helper 稳定后再重新启用。

## Open Questions

- 当前没有阻塞性的开放问题；如果实现期发现“一条皮带同时接收左右两侧 feeder”会显著破坏可读性，优先保持运行时串行支持，并把 richer visuals 作为后续增量变更处理。

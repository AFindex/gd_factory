## Context

`define-dual-industrial-standards` 与 `reframe-world-to-cabin-cargo-conversion` 已经把“世界大件不能直接跑在舱内轨道上”这件事做成了基础方向，但当前运行时仍然把大部分转换理解成“同一个 `FactoryItemKind` 只换 `CargoForm`”。结果是：

- 解包与封包仍然是 1:1 物品换形态，而不是“装箱清单 <-> 多个舱内小包”的真正转换。
- 世界大包与舱内小包的可视区别还不够稳定，世界侧仍然存在 2D 风格负载，和“重载大包”的定位冲突。
- 解包/封包舱段虽然已经更像处理舱，但还没有根据世界大包真实规格调节 footprint、舱体体量和处理动作。
- 目前没有正式约束“封包是否允许混装、混到什么程度、如何让玩家看懂货包模板”。

这次设计需要在不重写底层 belt / topology / transport 内核的前提下，把“大包层”和“小包层”彻底分开：世界大包只在重载节点间交接，舱内小包才进入普通物流网络。

## Goals / Non-Goals

**Goals:**
- 让世界大包以 manifest/模板的方式定义，不再等同于单个 `FactoryItemKind` 的放大版。
- 让解包从 `1 个世界大包 -> 多个舱内小包`，封包从 `多个舱内小包 -> 1 个世界大包`，并支持受控混装模板。
- 让世界大包在视觉上统一采用无阴影的 3D 贴图盒体，舱内小包继续保持 2D/billboard 风格。
- 让大包在舱内只存在于输入接口、解包舱、重载缓冲位、封包舱和输出接口之间，不进入普通 belt 拓扑。
- 让解包/封包舱段根据世界大包规格切换 footprint、占位体量和处理动作，并显式展示正在处理的大包。
- 让编辑器、world miniature、focused demo 与 smoke 都能读出“大包交接层 / 小包物流层”的差异。

**Non-Goals:**
- 不重写现有 `FactoryItem` 整体物流模拟或普通 belt 连接规则。
- 不把舱内也扩展成一整套重载输送网络；大包层只覆盖少量重载节点之间的交接。
- 不在这一轮做完全自由的“任意物品随便混装成任意世界包”系统。
- 不要求世界侧货包马上拥有完整 authored 3D 模型库；第一阶段允许统一走贴图盒体模板。

## Decisions

### 1. 世界大包采用“模板 + manifest”语义，而不是第二套完全独立的运行时物品类

运行时仍然沿用 `FactoryItem` 作为统一的物流 token，但对于 `WorldBulk / WorldPacked` 货物增加“世界货包模板”语义：货物需要能解析出自己的 `bundle template`、`size tier`、`manifest contents` 和 `presentation profile`。  
这样做的原因是当前项目的 transport、buffer、provider/receiver 链路已经广泛建立在 `FactoryItem` 上，如果直接引入另一套完全独立的 `WorldBundleItem` 运行时类型，会显著抬高迁移成本。

推荐实现方向：

- 新增 `WorldBundleTemplate` 配置层。
- 每个世界大包模板定义：
  - 输入/输出工业标准
  - 尺寸 tier
  - 贴图/标识
  - 解包清单
  - 封包模板类型（固定、类别、受控混装）
- `FactoryItem` 在世界货包形态下携带模板标识与 manifest 快照或模板引用。

备选方案是彻底引入独立的 `WorldBundle` 运行时类并让边界、缓冲、转换舱走另一套物品接口。这个方案在概念上更纯，但会立刻冲击现有 simulation API，因此放弃。

### 2. 大包在舱内只走“重载交接节点”，不进入普通传送带

设计上明确区分两层物流：

- `Heavy Handoff Layer`：世界大包使用，只存在于输入接口、解包舱、重载缓冲位、封包舱、输出接口。
- `Cabin Belt Layer`：舱内小包使用，继续沿用现有 belt / splitter / merger / bridge 拓扑。

这意味着大包在舱内不是“正常运输”，而是“节点间交接 / 暂存 / 处理”。  
这样做能彻底消除“大包怎么塞进舱内 belt”的矛盾，也能保持底层 belt 逻辑稳定。

实现上建议：

- 输入接口把世界大包挂在重载交接位。
- 解包舱直接从交接位拉入处理腔。
- 重载缓冲位只缓存世界大包，不参与普通 belt。
- 封包舱完成装箱后把成品大包挂到待发位。
- 输出接口从待发位接走大包。

备选方案是给大包再做一套完整的“重载轨道网络”。这个方案会迅速变成第二套内部物流系统，复杂度过高，因此只保留“局部重载交接节点”。

### 3. 解包输出采用“按清单逐步吐出”，封包采用“按模板累计装箱”

解包机一次只接受 1 个世界大包，但不会只产出 1 个舱内小包，而是根据模板清单按节拍持续吐出多个舱内小包。  
封包机则维护一个“装箱进度”，只有在清单满足目标模板时才生成 1 个世界大包。

推荐模板类型分三档：

- 固定模板：只能按精确配方装箱。
- 类别模板：允许某一类别物品按指定比例装箱，但不能随意跨类别。
- 受控混装模板：允许少量预定义清单混装，例如前线补给箱。

不允许的情况：

- 任意物品自由混装成无定义的世界大包。
- 不满足模板但强行输出“半成品世界包”。

备选方案是继续维持当前 1:1 逻辑，只通过时间和视觉做“好像拆了很多个”。这个方案会继续制造玩法与表现冲突，因此放弃。

### 4. 世界大包统一改成无阴影 3D 贴图盒体，舱内小包保持 2D/billboard

世界货包在视觉上要读成“重载包装件”，因此第一阶段统一采用：

- 纯 3D box mesh
- 贴图/标识贴面
- `CastShadow = Off`
- 可按模板尺寸 tier 改变盒体尺寸与比例

舱内小包继续使用轻量 2D/billboard 或轻量卡片式表现，以维持舱内物流的紧凑读感。  
这样能从材质、投影、体积和运动质感上把两类物流读成两套标准。

备选方案是世界侧继续保留 billboard/2D fallback，再靠尺寸解释是大包。这个方案在镜头较远时过于像“贴片图标”，不符合重载货包定位，因此放弃。

### 5. 解包/封包舱段 footprint 不做任意连续缩放，而做“规格 tier -> footprint 档位”映射

用户希望舱段能根据世界大包实际大小变化占位，但如果每个模板都任意改变 footprint，会让建造、校验、蓝图和 authored map 成本暴涨。  
因此设计上采用分档：

- `Compact`
- `Standard`
- `Wide`
- 必要时扩到 `Oversized`

每个世界大包模板绑定一个规格 tier，解包/封包舱段根据当前模板或当前配置的模板档位切换 footprint、预览尺寸和视觉机构伸展幅度。

这样既能体现“处理不同规格大包”的体量变化，又能让 footprint 仍然保持可验证、可 authored、可 smoke 的有限集合。

备选方案是完全动态按模型包围盒决定 footprint。这个方案会让 placement、blueprint 和 map validation 变得过于脆弱，因此放弃。

### 6. 解包/封包处理动画采用“机械臂 / 装卸机构 + 处理腔锚点”的组合

现有系统已经有 payload anchors，本次不推翻，而是扩展为：

- 重载挂架锚点
- 机械臂/夹持机构锚点
- 处理腔内大包锚点
- 出舱待发锚点

工作流程中：

- 大包在进入时显示在重载托架或挂架上。
- 机械臂或滑台把大包拉入舱体。
- 处理阶段显示局部打开、压缩、拆箱或装箱中的工作态。
- 解包阶段舱内 belt 侧按 manifest 节拍吐出多个小包。
- 封包阶段在小包累计完成后，处理腔重新出现完整成品大包。

备选方案是只在处理时把大包静态放在腔体中心。这个方案信息量太低，无法把“正在装卸 / 解包 / 封包”读出来，因此放弃。

## Risks / Trade-offs

- [模板与 manifest 配置量上升] → Mitigation: 第一阶段只覆盖 focused demo 实际使用的少数世界货包模板，并提供共享默认模板。
- [footprint 分档可能导致 authored map 调整量变大] → Mitigation: 限制到少量固定规格 tier，并为已有 focused / dual-standards 布局提供明确迁移步骤。
- [封包模板过于自由导致玩家不理解] → Mitigation: 第一阶段默认只开放固定模板，类别模板和混装模板只给少量明确命名的补给箱。
- [世界 3D 盒体太统一，失去资源辨识度] → Mitigation: 通过贴图、配色、尺寸比例、封条/标签和箱体细节保留模板识别。
- [重载节点与普通 belt 的双层语义在 UI 中仍然混淆] → Mitigation: 编辑器和 miniature 必须给重载节点单独的标识、颜色和提示语。
- [处理动画复杂度过高] → Mitigation: 第一阶段优先做锚点驱动的程序化机械臂/滑台动画，不要求立刻接入完整骨骼动画资源。

## Migration Plan

1. 先补 spec：定义 `WorldBundleTemplate`、manifest、封包模板边界、世界 3D / 舱内 2D 表现和重载节点规则。
2. 在数据层扩展世界货包模板与尺寸 tier，同时为 focused demo 使用的世界货包配置 manifest。
3. 重构 item visual profile，让世界货包统一走 3D textured box 且关闭阴影，舱内小包维持 2D/billboard。
4. 重构解包/封包/重载缓冲位和边界接口，让大包仅在重载节点中交接，并让舱段按尺寸 tier 切换 footprint 与体量。
5. 更新 interior authored map、editor preview、world miniature 和 focused demo 叙事。
6. 最后补 smoke/validation，覆盖 `1 -> N` 解包、模板封包、禁止大包进入舱内 belt、3D/2D 表现分离和 footprint 分档。

回滚策略：

- 如果 manifest 逻辑尚未稳定，可先只保留固定模板，不启用类别模板或受控混装模板。
- 如果动态 footprint 调整影响过大，可先保留标准档位，只对 visual scale 和 chamber animation 做规格区分。

## Open Questions

- 第一阶段是否需要让玩家在封包机 UI 中手动选择目标模板，还是针对 authored demo 先固定模板？
- `WorldBundleTemplate` 是否需要支持“拆箱损耗 / 装箱损耗”，还是先保持完全无损转换？
- world miniature 是否需要表现大包被机械臂拉入处理腔的完整动画，还是只显示关键锚点切换即可？

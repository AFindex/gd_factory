## Context

`define-dual-industrial-standards` 已经把世界工业标准与舱内工业标准在规则层做了解耦，但当前系统仍然没有把“世界货物比舱内货物大得多”这件事落实为稳定的视觉与交互契约。结果是：

- 世界侧货物和舱内料轨货物仍然容易被读成“同一物体缩小版”。
- 世界大件一旦进入舱内交接区域，当前方向仍容易被实现成“先缩小再处理”，这会削弱转换层存在的必要性。
- 解包机、封包机、缓冲模块仍然容易被读成普通小型物流设备，而不是跨工业标准的转换舱。
- 玩家很难从空间上理解为什么世界侧货物不能直接流进舱内料轨。
- 舱内模块的体量仍然偏向“缩小版世界建筑”，削弱了舱段维护空间的可信度。

这次变更不打算推翻已经成立的共通物流语法。底层 belt 拓扑、端口交互、连接规则和放置交互仍然共享。真正要重塑的是世界标准与舱内标准之间的“载荷尺度断层”以及由此带来的转换流程、模型表现和 demo 叙事。

同时，这次改动需要兼容已有方向：

- 世界侧仍然运输大件外部货物。
- 舱内仍然保持真实可进入的维护空间，而不是纯抽象界面。
- 内外物流在底层逻辑上继续复用同一套 transport / topology / interaction 体系。
- 主要差异应体现在配置、可视载荷、结构轮廓、尺寸关系和玩法节拍上。

## Goals / Non-Goals

**Goals:**
- 明确建立两种货物尺度标准：世界侧为大件载荷，舱内为小型标准载具。
- 明确规定世界大件即使进入内部工厂，也只会以原尺寸出现在交接/转换/暂存位置，而不会预先缩小。
- 明确规定世界大件不能直接进入舱内料轨，必须经过单件解包/封包转换流程。
- 将解包机与封包机重定义为一进一出、一次处理一个世界大件的转换舱，并在模型上展示当前处理的载荷。
- 将缓冲模块重定义为服务于大件转换节拍的暂存/等待结构，而不是泛化的小型加工机。
- 调整舱内模块体量，让其接近一个世界大件货物的尺寸，使空间关系更可信。
- 让编辑器、边界接口和 demo 都能清楚教会玩家“世界大件 -> 舱内小载具 -> 世界大件”的链条。

**Non-Goals:**
- 不重写底层物流模拟，不引入第二套 belt / topology / routing 核心。
- 不在这次变更中重构完整资源体系或所有配方平衡。
- 不要求世界与舱内共享同一套建筑模型后再靠缩放解决问题。
- 不把舱内空间做成纯 UI，也不追求世界建筑与舱内模块的 1:1 写实重建。

## Decisions

### 1. 将世界货物与舱内货物定义为不同的载荷标准，而不是同资源的缩放版本

系统继续保留资源语义共通，但在表现与空间占用上明确分为两类载荷：

- `WorldCargo`: 世界侧大件载荷，只出现在世界 belt、世界 depot、边界交接口、转换舱处理腔体和大件暂存位中，并始终保持同一尺寸标准。
- `CabinCarrier`: 舱内标准载具，只出现在舱内料轨、舱内缓冲结构和舱内生产模块之间。

这样设计的原因是，用户最在意的问题不是“颜色不一样”，而是“直觉上放不下”。因此必须把放不下这件事做成系统事实，而不是仅仅在美术上暗示。

备选方案是继续沿用同一货物表示，只通过缩放与材质区分内外。这个方案虽然实现成本低，但会继续制造“为什么不能直接进舱内”的违和感，因此放弃。

### 2. 解包机与封包机采用单件节拍的转换舱模型

解包机与封包机不再被表达为多口小型物流节点，而是单件处理腔体：

- 解包机：`1 world input -> 1 cabin output lane`
- 封包机：`1 cabin input lane -> 1 world output`
- 每次只处理一个世界大件。
- 工作时必须在模型上显示被处理的世界大件或装填中的成品大件。
- 世界大件在进入舱内处理区域后仍保持世界标准尺寸，而不是先被缩小后再展示。

这种设计的重点不是限制 throughput，而是让视觉、功能和空间逻辑统一。玩家看到一个世界大件进入处理舱，随后转换成可在舱内轨道流动的小载具，直觉上会更容易接受。

备选方案是保留多输入多输出接口，只在文案上说它是转换舱。这个方案会导致转换建筑继续像普通物流节点，削弱存在必要性，因此放弃。

### 3. 缓冲模块定位为大件转换节拍的暂存结构，而不是通用加工设备

缓冲模块的职责不是“又一个加工机”，而是围绕单件大载荷转换链路提供节拍整理：

- 在解包前充当待处理大件的进站暂存位。
- 在封包前充当待出站成品的集结位。
- 在视觉上优先表现为托架、停泊位、夹持座、缓存架或装载台。
- 暂存中的世界大件继续保持世界标准尺寸，以便玩家理解这里不是普通舱内仓位。

这样做能让缓冲与解包/封包形成明确分工，也能解释为什么内部虽小，仍然需要围绕大件交换做节拍管理。

备选方案是维持旧的缓冲小箱体/小设备语义。这样会让缓冲、封包、解包在视觉和玩法上过度同质，因此放弃。

### 4. 舱内模块体量以“接近一个世界大件”作为基准，而不是继续走微缩工厂路线

舱内模块的大小不再参考“世界建筑缩小版”，而参考“一个世界大件货物 + 人员维护空间”：

- 单个舱内模块大约与一个世界大件同量级。
- 玩家可以相信大件在模块之间被接收、拆开、处理、重新装填。
- 舱内料轨只承担小载具流动，因此可以嵌入地槽、壁槽或设备底盘。

这能同时调和主角尺度与货物尺度：主角不需要缩放，舱内也不会变成小人国；大的东西进入的是模块腔体，小的东西才在舱内轨道上跑。

备选方案是继续把舱内建筑做得比世界货物小很多，再靠镜头或文案解释。这会直接重现用户指出的视觉冲突，因此放弃。

### 5. 边界接口只负责大件交接，不表现为“世界货物直接上舱内轨”

边界接口的语义需要调整为：

- 世界大件只在世界物流网络、壳体边界接口和转换舱之间交接。
- 世界大件即使进入壳体内侧的交接区或转换舱，也继续保持原尺寸标准。
- 舱内轨道从转换舱之后才开始出现。
- 任何预览、world miniature、demo 场景都不能把世界大件直接画成在舱内 feed rail 上滑行。

这样能把边界附件从普通 port 提升为标准切换边界，也能让部署后的世界表现更合理。

备选方案是让边界附件直接贴着舱内 belt 出入货物。这样虽然复用了更多现成表现，但会把“转换层”视觉上消掉，因此放弃。

### 6. 底层逻辑保持共通，实现差异主要落在配置与 presentation 层

虽然这次 change 会明显改变玩家感知，但技术上不需要拆底层：

- 物流连接、路径、阻塞、交互和放置规则继续共享。
- 差异通过 item visual profile、structure visual profile、conversion-building 配置和 editor/demo 文案来表达。
- 解包/封包/缓冲的玩法差异主要体现为端口限制、处理节拍和可见载荷状态，而不是重造 transport 内核。

这样既能满足新的视觉/语义要求，也能避免把 change 扩张成难以落地的大规模重构。

## Risks / Trade-offs

- [单件转换节拍让玩家担心吞吐过低] → Mitigation: 允许后续通过升级速度、并列转换舱、缓存编组来扩展效率，但首阶段先保证语义清晰。
- [世界大件与舱内小载具区分不够强，最后仍像缩放版本] → Mitigation: spec 明确要求不同轮廓、载体、占位关系和处理腔可见性，不接受单纯缩放换皮。
- [为了“塞进舱内”又把世界大件缩小显示] → Mitigation: 通过模块尺度、处理腔和暂存位去吸收空间需求，spec 明确禁止把世界大件在舱内交接区预先缩成 cabin cargo。
- [舱内模块做大后，内部空间容易变挤] → Mitigation: 让料轨嵌入模块之间的槽道，并把人行维护空间与物流层错开，而不是把所有元素堆在同一平面。
- [缓冲模块定位变化后，玩家一开始不明白它和仓储/解包的区别] → Mitigation: 在模型轮廓、命名、动画反馈和编辑器提示中强调“节拍暂存位”而非普通仓储。
- [底层逻辑不改，表现层却要强行讲述尺度断层，容易出现漏网之鱼] → Mitigation: 在 demo、editor preview 和 validation 中都增加针对“大件不能直接上舱内轨”的约束验证。

## Migration Plan

1. 先补齐 spec，明确世界大件、舱内小载具、转换舱、缓冲位和边界接口的契约。
2. 再调整 item / structure visual profile 的配置能力，使同一资源和结构能根据工业标准落到不同 presentation。
3. 之后重构解包机、封包机、缓冲模块和舱内关键模块的模型语言与可视载荷状态。
4. 更新内部编辑预览、world miniature 与 focused demo，让演示链条清晰可读。
5. 最后补 smoke / validation，避免后续回归成“世界货物进舱就先缩小”或“世界货物直接在舱内轨道上跑”。

回滚策略：

- 如果某些高保真表现暂时做不完，可以先用程序化占位，但仍必须保留大件处理腔和小载具轨道的尺度差异。
- 若 demo 调整过大影响当前回归，可先保留旧流程作为对照场景，但新 focused demo 必须体现新的转换语义。

## Open Questions

- 舱内小载具首阶段是否只区分 `InteriorFeed` 一类，还是同时分出弹药/液料等专用 carrier family？
- 解包与封包处理中显示的是完整世界货物模型，还是“进入腔体后局部拆解/组装中的过渡态”？
- world miniature 中是否需要完整显示大件进入转换舱的动画，还是保持更抽象但仍不违背尺度语义的表现？
